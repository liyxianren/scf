## 一、项目背景理解

### 1.1 业务现状

我们公司的核心业务是为有留学需求的中国高中生提供**定制化科创项目服务**。服务模式是根据学生的个人经历和家庭背景，定制符合其"人设"的科研项目，帮助他们在申请海外院校时展现独特的竞争力。

### 1.2 当前痛点

目前项目交付物存在一个核心问题：**学生完成项目后，只能获得一张公司颁发的纸质证书**。

这种证书在实际申请中存在以下局限：
- 公信力有限，无法与学校官方或权威机构的认证相比
- 无法体现学生在项目中的真实能力和工程素养
- 难以向招生官证明学生是"深度参与"还是"表面完成"

### 1.3 解决方向

引入**"工程手册"（Engineering Portfolio）**作为新的交付物形式，通过结构化、专业化的技术文档，将学生的项目过程和成果转化为具有说服力的申请材料。

## 二、工程手册的价值主张

### 2.1 为什么工程手册能提升项目竞争力？

传统项目交付物（证书、代码、演示视频）的问题：

| 传统交付物 | 存在的问题 |
|-----------|-----------|
| 纸质证书 | 公信力不足，无法体现技术深度 |
| 代码仓库 | 信息过载，招生官难以快速评估 |
| 演示视频 | 只展示结果，不体现过程 |

**工程手册的解决方式**：将隐性的工程思维转化为显性的、可评估的文档证据，展示学生"发现问题→分析问题→解决问题"的完整思维链路。

### 2.2 带给学生的具体价值

**价值一：可信度提升**

通过文档中的技术细节、迭代记录、测试数据等，招生官可以判断学生是真正理解了技术原理，还是仅仅"调用了别人的库"。

**价值二：差异化竞争**

当大多数申请者只能提供"我做了一个项目"的笼统描述时，一份专业的工程手册能让学生从"有项目经历"升级为"具备工程师思维"。

**价值三：面试支撑**

工程手册可以成为面试时的"证据库"。当教授问到技术细节时，学生可以直接引用文档中的具体章节来回答，而不是临时组织语言。

**价值四：元能力证明**

撰写技术文档本身就是一种重要的工程能力。这份手册同时证明了学生的技术沟通能力和项目管理能力。

## 三、当前 DEMO 的问题诊断

基于"肠音慧眼"项目模板的分析，我们识别出以下需要改进的问题：

### 3.1 形式规范性不足

当前模板缺少标准工程文档应有的元素：
- 无版本控制信息
- 无修订历史记录
- 无术语定义表
- 无标准的文档编号系统

这些元素在国际工程教育认证标准（如 ABET、CDIO）中是衡量文档专业度的重要指标。

### 3.2 技术描述深度不够

当前问题示例：
> "采用 PyTorch 构建深度神经网络（如 CNNs, RNNs 或 LSTMs）"

这类表述过于泛化，无法体现学生对技术的真正理解。

**应该改进为**：说明"为什么选择这个技术"、"考虑过哪些替代方案"、"做了什么权衡取舍"。

### 3.3 证据链条不完整

模板中的素材收集建议与文档内容之间缺乏强绑定。理想状态下，文档中的每一个技术声明都应该有对应的证据支撑（截图、代码片段、测试数据等）。

### 3.4 体系针对性缺失

当前模板没有考虑美国、英国、港新等不同申请体系对技术文档的偏好差异。同一份材料无法在所有体系中都达到最优效果。

## 四、工程手册内容结构规划

### 4.1 整体结构概览

```
工程手册
├── 元数据
├── 执行摘要
├── 问题定义与需求分析
├── 系统设计
├── 实现与迭代（核心章节）
├── 测试与验证
├── 反思与展望
└── 附录
```

### 4.2 各章节详细说明

#### 元数据部分

这是文档的"身份证"，包含以下内容：

| 元素 | 说明 | 示例 |
|-----|------|-----|
| 项目名称 | 中英文双语 | 肠音慧眼 / GutSound AI |
| 版本号 | 语义化版本 | v1.2.0 |
| 作者信息 | 学生姓名及角色 | 张三（算法开发） |
| 完成日期 | 项目完成时间 | 2026年1月 |
| 修订历史 | 重要版本变更记录 | 见下方表格 |
| 术语表 | 专业术语解释 | MFCC: 梅尔频率倒谱系数 |

#### 执行摘要（1页以内）

这是整份文档的"电梯演讲"，必须包含：

- **问题陈述**：用一句话说明要解决什么问题
- **解决方案概述**：用一句话概述技术方案
- **关键技术指标**：3-5个量化结果（如准确率、响应时间等）
- **个人贡献声明**：明确说明学生本人在项目中的具体贡献

#### 问题定义与需求分析

展示学生"发现问题"的能力：

**背景调研**
- 这个问题为什么重要？
- 现有解决方案有什么不足？
- 相关领域的技术发展现状如何？

**利益相关者分析**
- 这个项目服务于谁？
- 他们的核心诉求是什么？
- 存在哪些约束条件？

**需求规格说明**
- 功能性需求（系统必须做什么）
- 非功能性需求（性能、安全、可用性等）

**成功标准定义**
- 如何判断项目是否成功？
- 需要可量化的评估指标

#### 系统设计

展示学生"设计方案"的能力：

**技术选型与理由**
- 选择了什么技术栈？
- 为什么选择它而不是其他方案？
- 做了哪些技术调研和对比？

**系统架构设计**
- 系统的整体结构图
- 模块划分及职责说明
- 模块间的依赖关系

**数据流与交互逻辑**
- 数据如何在系统中流动？
- 用户交互的时序流程
- 关键状态的转换逻辑

**接口规范**（如适用）
- API 接口定义
- 数据格式说明
- 错误处理机制

#### 实现与迭代（核心章节）

这是整份文档的**重点章节**，展示学生"解决问题"的能力：

**开发环境与工具链**
- 编程语言及版本
- 框架和库的选择
- 开发工具和协作平台

**核心模块实现**

每个模块按照统一结构描述：
1. 遇到什么问题（Challenge）
2. 设计什么方案（Approach）
3. 如何实现（Implementation）
4. 效果如何（Result）

**关键技术挑战与解决方案**

重点记录那些不是一次就成功的尝试：
- 遇到了什么困难？
- 尝试了哪些方案？
- 最终如何解决？
- 从中学到了什么？

**版本迭代记录**
- 重要的版本变更
- 改进的理由和效果

#### 测试与验证

展示学生"验证结果"的能力：

**测试策略说明**
- 采用什么测试方法？
- 为什么选择这种方法？

**测试用例与执行结果**
- 关键测试用例设计
- 测试执行记录
- 缺陷发现与修复

**性能基准测试**（如适用）
- 测试环境说明
- 性能指标定义
- 测试结果数据
- 与基准的对比分析

**用户测试反馈**（如有）
- 测试用户画像
- 反馈收集方法
- 主要反馈内容及响应

#### 反思与展望

展示学生的"元认知"能力：

**工程经验总结**
- 通过这个项目学到了什么？
- 哪些决策是正确的？
- 如果重新开始会怎么做？

**技术债务与已知限制**
- 诚实地说明项目还有哪些不足
- 存在哪些已知的技术债务
- 哪些功能因时间限制未能实现

**未来改进方向**
- 如果有更多时间和资源，会如何优化？
- 有哪些可以继续探索的方向？

#### 附录

补充支撑材料：

- **附录A**：完整代码仓库链接
- **附录B**：关键代码片段（带详细注释）
- **附录C**：原始数据样本
- **附录D**：参考文献列表

## 五、不同申请体系的差异化策略

### 5.1 美国体系

**适用范围**：Common App、Coalition App 等美本申请

**核心关注点**：这个项目如何体现申请者是一个怎样的人

**文档调整策略**：

| 调整维度 | 具体做法 |
|---------|---------|
| 执行摘要 | 增加个人动机描述（为什么对这个问题感兴趣） |
| 反思展望 | 详细展开，强调个人成长和对未来的思考 |
| 利益相关者 | 加入项目对社区或用户的实际影响 |
| 语言风格 | 适度采用第一人称叙事，建立情感连接 |
| 总体篇幅 | 15-20页，可视化内容比例较高 |

### 5.2 英国体系

**适用范围**：UCAS Personal Statement 配套材料

**核心关注点**：学生对所申请学科的学术热情和知识深度

**文档调整策略**：

| 调整维度 | 具体做法 |
|---------|---------|
| 背景调研 | 更加详实，展示对领域现状的深入了解 |
| 技术选型 | 体现学术思考，可引用相关论文或技术文献 |
| 参考文献 | 必须规范完整，格式符合学术标准 |
| 语言风格 | 学术化、客观，避免过多个人情感表达 |
| 总体篇幅 | 10-15页，以文字论述为主 |

### 5.3 港新体系

**适用范围**：香港、新加坡院校本科申请

**核心关注点**：技术硬实力和工程规范性

**文档调整策略**：

| 调整维度 | 具体做法 |
|---------|---------|
| 系统设计 | 非常详细，包括完整的架构图和数据流图 |
| 测试验证 | 必须有充分的量化数据支撑 |
| 性能基准 | 有明确的对比基准和测试方法说明 |
| 代码展示 | 更多地展示代码片段，体现编程规范 |
| 总体篇幅 | 20-30页，技术细节完整呈现 |

### 5.4 体系对比总结

| 维度 | 美国体系 | 英国体系 | 港新体系 |
|------|---------|---------|---------|
| 核心关注 | 个人特质与成长 | 学术热情与深度 | 技术实力与规范 |
| 重点章节 | 执行摘要、反思展望 | 背景调研、技术选型 | 系统设计、测试验证 |
| 语言风格 | 叙事化、情感连接 | 学术化、客观严谨 | 技术化、数据支撑 |
| 建议篇幅 | 15-20页 | 10-15页 | 20-30页 |

## 六、系统输入与输出定义

### 6.1 用户输入

**必选项**

| 输入项 | 说明 | 格式要求 |
|-------|------|---------|
| 项目说明文档 | 介绍项目背景、目标、使用的技术和方法 | Markdown / Word / PDF |
| 目标申请体系 | 美国 / 英国 / 港新（可多选） | 单选或多选 |

**可选项**

| 输入项 | 说明 | 格式要求 |
|-------|------|---------|
| 源代码 | GitHub 仓库链接或代码压缩包 | URL / ZIP |
| 过程素材 | 截图、测试数据、迭代记录等 | 图片 / 表格 / 文档 |

### 6.2 系统输出

**主要交付物**

- 完整的工程手册文档（Markdown / Word / PDF 格式）
- 如选择多个申请体系，则生成多个针对性版本

**辅助交付物**

- 素材补充清单：提示用户还可以补充哪些材料来进一步提升文档质量

## 七、待确认事项

为确保我们的理解与业务需求一致，以下问题需要进一步确认：

### 7.1 关于体系覆盖范围

除了美国、英国、港新之外，是否还需要考虑其他申请体系？
- 加拿大体系
- 澳洲体系
- 欧洲体系（如荷兰、德国等）

### 7.2 关于文档语言

工程手册的语言版本如何安排？
- 统一使用英文撰写
- 根据目标体系提供中英文版本
- 提供中文版本 + 英文摘要

### 7.3 关于技术导师配合

- 当前的素材收集流程是否已经标准化？
- 技术导师在教学过程中是否有固定的节点来收集学生的过程材料？
- 素材收集指南是否需要同步更新？

### 7.4 关于自动化程度

Agent 系统的定位是什么？
- 完全自动生成初稿，人工只做审核
- 生成框架和部分内容，人工补充关键细节
- 作为辅助工具，主要由人工撰写

对于技术含量较高的章节（如核心模块实现），是否需要技术导师的二次审核和补充？

### 7.5 关于与现有流程的整合

工程手册的生成时机是什么？
- 项目结束时一次性生成
- 伴随项目进度实时更新
- 分阶段生成，最后合并

## 八、下一步工作计划

待需求对齐确认后，建议按以下顺序推进：

1. **确定各节点的详细设计**：明确每个 Agent 节点的输入、输出、处理逻辑
2. **设计 Prompt 模板**：针对 ChatGLM 的能力特点，设计每个节点的系统提示词
3. **制定素材收集规范**：与技术导师的教学流程对齐，确保能获取高质量原始素材
4. **开发原型系统**：先实现单一体系的完整流程，验证可行性
5. **扩展多体系支持**：在验证可行性后，扩展支持其他申请体系
