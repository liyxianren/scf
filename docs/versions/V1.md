# V1 当前产品基线

更新时间：2026-03-26

## 一句话定义

V1 已经不是单纯的编程教学网站，而是一个以 `报名 intake -> 排课 -> 工作流协作 -> 反馈闭环 -> OA 执行` 为主干、并开始具备学生优先 AI 排课与业务风险兜底能力的 SCF 内部交付平台。

## V1 已经具备的核心能力

### 1. 角色与账号体系

- 已实现 `admin / teacher / student` 三类角色。
- 已实现登录、角色跳转、角色面板和对应 API。
- 当前主入口不再只是教学页面，而是包含内部教务和执行角色的操作界面。

主要证据：

- `modules/auth/models.py`
- `modules/auth/routes.py`

### 2. Enrollment 与学生 intake 基线

- Admin 可以创建 `Enrollment`。
- 学生通过 token 化 intake 链接提交信息。
- 系统会创建或绑定学生账号、学生档案。
- 老师可维护 availability，Admin 可自动匹配、确认方案，学生可确认或拒绝。
- 当前状态流覆盖：
  - `pending_info`
  - `pending_schedule`
  - `pending_student_confirm`
  - `confirmed`
  - `active`
  - `completed`

主要证据：

- `modules/auth/models.py`
- `modules/auth/enrollment_routes.py`
- `modules/auth/services.py`
- `tests/integration/test_enrollment_flow.py`

### 3. 学生优先排课 AI 基座

- Enrollment 已支持排课约束字段：
  - `sessions_per_week`
  - `delivery_urgency`
  - `target_finish_date`
- 学生 intake 已支持“自然语言文本 + 粘贴截图/聊天文本”的时间表达，并保留结构化时段 fallback。
- 系统会把学生输入解析并落库为：
  - `availability_intake`
  - `candidate_slot_pool`
  - `recommended_bundle`
  - `risk_assessment`
- 当前排课策略已支持：
  - 按 `sessions_per_week` 做每周硬约束
  - 先做候选池与风险评估，再决定是否进入老师提案
  - 低风险方案直通学生确认
  - 高风险方案回教务处理

主要证据：

- `modules/auth/availability_ai_services.py`
- `modules/auth/models.py`
- `modules/auth/services.py`
- `modules/auth/workflow_services.py`
- `tests/integration/test_enrollment_flow.py`

### 4. 工作流待办层与业务容灾兜底

- 系统已经不是只有普通 todo。
- 已经有基于业务语义的 workflow todo：
  - `enrollment_replan`
  - `leave_makeup`
  - `schedule_feedback`
- 已支持老师提案、教务审核/发送给学生、学生确认/拒绝等动作。
- 失效业务对象或被删除请求不会继续暴露成可点击空页面的孤儿待办。
- 教务 action center 已能暴露 `scheduling_risk_cases`，把排课风险与普通待办分开。

主要证据：

- `modules/auth/workflow_routes.py`
- `modules/auth/workflow_services.py`
- `modules/auth/routes.py`
- `modules/oa/routes.py`
- `tests/integration/test_deletion_flows.py`
- `tests/integration/test_action_centers.py`
- `tests/integration/test_enrollment_flow.py`

### 5. 学生 OA 任务中心

- 学生端已开始从“流程/案件视角”收敛到“任务视角”。
- 学生 action center 已统一返回：
  - `student_action_items`
  - `student_tracking_items`
- 学生确认/退回已提供统一动作入口：
  - `/auth/api/student-actions/confirm`
  - `/auth/api/student-actions/reject`
- 新学生端主交互不再把 `enrollment` / `workflow-todo` 双对象模型直接暴露成两套动作入口。

主要证据：

- `modules/auth/routes.py`
- `modules/auth/enrollment_routes.py`
- `modules/auth/templates/auth/student_dashboard.html`
- `tests/integration/test_action_centers.py`
- `tests/integration/test_auth_access.py`
- `tests/integration/test_enrollment_flow.py`

### 6. OA 排课与课后反馈

- Admin 可维护 `CourseSchedule`，支持日历视图和 CRUD。
- 课次可绑定报名记录，并驱动报名状态同步。
- 课后反馈已建模为 `CourseFeedback`，可参与结课完成判断。
- 请假、补课、反馈之间已有联动约束。

主要证据：

- `modules/oa/models.py`
- `modules/oa/routes.py`
- `modules/auth/enrollment_routes.py`
- `tests/integration/test_oa_routes.py`

### 7. Excel 课表导入

- Excel 导入并非占位功能，而是带有导入记录、语义解析、冲突检测、待办生成的实流程。
- 已经存在 `ScheduleImportRun`，并且近期仍在持续修复导入与绑定细节。

主要证据：

- `modules/oa/models.py`
- `modules/oa/services.py`
- `tests/integration/test_oa_p1_regressions.py`

### 8. 内部聊天与协作

- 已支持角色间聊天、未读数、会话列表。
- 排课拒绝反馈已能回流到运营处理链路，不再只是孤立消息。

主要证据：

- `modules/auth/chat_routes.py`
- `modules/auth/models.py`
- `tests/integration/test_enrollment_flow.py`

### 9. 周边已实现模块

- `education/`
  - 编程学习、题目、代码运行。
- `agents/`
  - 创意项目生成与计划书生成。
- `handbook/`
  - 工程手册生成与导出。

这些模块都是真实存在的产品能力，但它们目前不是仓库演进的主重心。

## V1 当前边界

以下内容 **不属于 V1 已完成能力**：

- 真正的图片上传 OCR、多模态课表识别、语音/视频时间解析
- 老师端完整的“候选池可视化选课”专用交互
- 独立的基础设施级容灾、灾备切换、高可用治理
- 显式的项目交付管理对象
- `ProjectTrack / SubProject / Milestone / Risk Board`
- 售前学生档案 / 线索管理 CRM

当前 V1 所说的“容灾”，主要指 **业务流程层的异常兜底与风险回退**，不是基础设施级 DR。

## V1 当前现实判断

- V1 的 `auth + enrollment + workflow + oa + feedback + chat` 基站已经完成，当前不是从零搭平台，而是在既有交付链路上继续做 AI 增效与异常兜底。
- 学生侧的首要目标已经从“让学生理解流程”切换成“让学生更快表达时间、更快确认方案、更少跳页找动作”。
- 当前已经落地的是 AI 排课基座，不是最终形态：
  - 已有文本解析、候选池、风险评估、统一学生动作入口
  - 尚未完成真正多模态解析、老师候选池 UI、请假/补课全链路复用
