# V1 当前产品基线

更新时间：2026-03-21

## 一句话定义

V1 已经不是单纯的编程教学网站，而是一个以 `报名 intake -> 排课 -> 工作流协作 -> 反馈闭环 -> OA 执行` 为核心的 SCF 内部交付平台。

## V1 已经具备的核心能力

### 1. 角色与账号体系

- 已实现 `admin / teacher / student` 三类角色。
- 已实现登录、角色跳转、角色面板和对应 API。
- 当前主入口不再只是教学页面，而是包含内部教务和执行角色的操作界面。

主要证据：

- `modules/auth/models.py`
- `modules/auth/routes.py`

### 2. Enrollment 与学生 intake 流程

- Admin 可以创建 `Enrollment`。
- 学生通过 token 化 intake 链接提交信息。
- 系统会创建或绑定学生账号、学生档案。
- 老师可维护可用时间，Admin 可自动匹配、确认方案，学生可确认或拒绝。
- 当前状态流覆盖：
  - `pending_info`
  - `pending_schedule`
  - `pending_student_confirm`
  - `confirmed`
  - `active`
  - `completed`

主要证据：

- `modules/auth/models.py`
- `modules/auth/enrollment_routes.py`
- `modules/auth/services.py`
- 提交记录：`7e50cb6`

### 3. 工作流待办层

- 系统已经不是只有普通 todo。
- 已经有基于业务语义的 workflow todo：
  - `enrollment_replan`
  - `leave_makeup`
  - `schedule_feedback`
- 已支持老师提案、教务审核/发送给学生、学生确认/拒绝等动作。

主要证据：

- `modules/auth/workflow_routes.py`
- `modules/auth/workflow_services.py`
- `modules/oa/models.py`
- 提交记录：`828402f`、`92e6b45`

### 4. OA 排课与课后反馈

- Admin 可维护 `CourseSchedule`，支持日历视图和 CRUD。
- 课次可绑定报名记录，并驱动报名状态同步。
- 课后反馈已建模为 `CourseFeedback`，可参与结课完成判断。
- 请假、补课、反馈之间已有联动约束。

主要证据：

- `modules/oa/models.py`
- `modules/oa/routes.py`
- `modules/auth/enrollment_routes.py`

### 5. Excel 课表导入

- Excel 导入并非占位功能，而是带有导入记录、语义解析、冲突检测、待办生成的实流程。
- 已经存在 `ScheduleImportRun`，并且近期仍在持续修复导入与绑定细节。

主要证据：

- `modules/oa/models.py`
- `modules/oa/services.py`
- 提交记录：`c7fdcd7`、`0fbd875`、`39d4602`

### 6. 内部聊天与协作

- 已支持角色间聊天、未读数、会话列表。
- 排课拒绝反馈已能回流到运营处理链路，不再只是孤立消息。

主要证据：

- `modules/auth/chat_routes.py`
- `modules/auth/models.py`
- `tests/integration/test_enrollment_flow.py`

### 7. 周边已实现模块

- `education/`
  - 编程学习、题目、代码运行。
- `agents/`
  - 创意项目生成与计划书生成。
- `handbook/`
  - 工程手册生成与导出。

这些模块都是真实存在的产品能力，但它们目前不是仓库演进的主重心。

## V1 当前边界

以下内容 **不属于 V1 已完成能力**：

- 显式的项目交付管理对象
- `ProjectTrack / SubProject / Milestone / Risk Board`
- 以节点和风险预警为中心的交付看板
- 售前学生档案 / 线索管理 CRM

当前主业务对象仍然是 `Enrollment`，它更像已确认服务的起点，而不是完整的项目交付对象。

## V1 当前现实判断

- V1 已经可用于内部教务与执行管理。
- OA P1 相关流程已经可用，但仍处于稳定化和回归修补阶段。
- 文档体系此前明显落后于代码现实，因此本次新增 `docs/` 和 repo-local skills 作为长期交接底座。
