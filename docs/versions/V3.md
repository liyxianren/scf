# V3 目标：销售接入与全链路外部化平台

更新时间：2026-03-23

## V3 的定位

V3 建立在 V2 的网站母端口、项目交付管理和提醒外化能力之上，继续把 SCF 的完整服务链路平台化。它不再只关注“内部怎么执行”，而是开始回答“销售怎么接入、外部怎么协作、整条服务链如何统一到一个平台里”。

## V3 与 V2 的默认边界

- `V2`
  - 重点是网站母端口保留、交付管理、提醒外化、内部执行效率
- `V3`
  - 重点是销售 / 售前接入、全链路平台化、外部角色协作面和更完整的对外产品层

如果 leadership 决定把一部分销售入口提前拉进 V2，也应理解为“把 V3 的轻量子集前置”，而不是改变 V3 的整体定位。

## V3 默认方向

### 1. 销售 / 售前接入

- 引入早于 `Enrollment` 的学生档案、线索状态和销售推进面。
- 让销售、顾问、交付之间有正式 handoff，而不是靠聊天和表格转交。
- 典型状态可以包括：
  - 沟通中
  - 需求梳理中
  - 方案制定中
  - 待报名
  - 已转报名
  - 已转交付

### 2. 服务全链路贯通

- 把售前、已报名服务、项目交付、排课、反馈、结课输出串成一条连续主线。
- 让一个学生从初次接触到结课归档的长期服务历史可以被完整回溯。

### 3. 多渠道外部协作面

- 内部渠道不再只是网站，可深入接入 `OpenClaw + Feishu`。
- 外部渠道不再只是“被提醒一次”，而是逐步形成学生 / 家长 / 未来合作方可使用的协作面。
- 网站仍是母端口，外部渠道是镜像入口、动作入口或提醒入口，不是状态 owner。

### 4. 渠道编排与通知平台化

- 把网站、OpenClaw、Feishu、未来微信 / 短信 / 邮件 / 小程序等放进同一个渠道编排框架中。
- 统一管理：
  - 哪类事件发给谁
  - 通过什么渠道发
  - 失败如何补发
  - 哪些动作回写到网站业务层

### 5. 更完整的输出物与归档

- 结课文书
- 进程文档
- 代码归档
- 拍摄资料
- 比赛材料
- 交付总结与长期服务档案

### 6. 更明确的多角色产品层

- 销售 / 顾问视角：线索、需求、转报名、转交付
- Admin / 教务视角：运营调度、交付进度、风险面板
- 老板视角：全局风险、节点到期、阶段分布、团队吞吐
- 执行人视角：我负责的项目、我认领的节点、我即将到期的任务
- 外部学生 / 家长视角：提醒、确认、结果查看、服务进度感知

### 7. 外部产品化准备

- 为未来的小程序、飞书、通知渠道、潜在产品化版本保留稳定边界。
- 但仍坚持站内数据库与业务层为主事实源，不让外围渠道反客为主。

## V3 当前默认原则

- 先做真实业务链的平台化，不先做漂亮但空心的 SaaS 壳。
- 先把销售、交付、提醒、归档串起来，再谈复杂 BI、租户化和大规模产品包装。
- 所有渠道集成都应建立在网站业务层之上，不允许外部渠道直接成为业务真相源。

## V3 先不锁死的内容

以下内容在 V3 只保留方向，不提前锁死实现：

- 销售前端最终 UI 形态
- 学生 / 家长外部端的具体产品形态
- 第三方集成细节
- 多租户或对外 SaaS 化细节
- 复杂 BI / 报表系统实现方式
- 最终采用哪些外部渠道作为主学生触达面

## V3 的意义

V3 的目标不是让系统看起来更“大”，而是让 SCF 的真实服务模式从“内部执行系统”升级为一套可持续复用、可交接、可扩展、可对外延展的平台能力。
