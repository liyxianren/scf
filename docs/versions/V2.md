# V2 目标：交付管理、学生触达与课后反馈自动化

更新时间：2026-03-24

## 一句话定义

V2 当前首要任务不再是单独推进内部 `OpenClaw + Feishu` 外化，而是在保持网站业务层与 workflow-safe API 为唯一真相源的前提下，先完成 `短信提醒接入 + 销售前端接入 + 腾讯会议课后反馈接入` 三条开发主线，再在此基础上继续推进内部外化入口与项目交付管理层。

## V2 现在要解决什么

V1 已经具备站内执行闭环，但当前仍有四个直接缺口：

- 学生侧缺少低摩擦、可规模化落地的主动提醒通道
- 销售前端的沟通记录仍游离在系统外，无法顺滑进入学生档案和录取流程
- 课后反馈仍高度依赖老师手写，会议录制与转写还没有进入标准反馈链路
- 内部团队的站外提醒与动作入口仍有价值，但不再是当前唯一主线

这会带来直接问题：

- 学生提醒仍依赖人工触达或临时脚本，难以形成稳定投递和审计
- 销售沟通内容无法自然沉淀为“学生画像 -> 学生档案草稿 -> Enrollment/录取”
- 课后反馈产出慢、主观波动大，课堂证据无法被系统复用
- 后续无论接微信、小程序、OpenClaw、Feishu 还是其他渠道，都缺少一套已经被业务验证的统一入口

## V2 当前默认目标顺序

### 1. 保住网站母端口

- 网站业务层和数据库继续是单一事实源。
- 当前 `auth + enrollment + workflow + oa + feedback + chat` 接口语义要保留。
- 短信、腾讯会议、未来微信/小程序、OpenClaw、Feishu 都只能通过网站业务层或 workflow-safe API 接入，不能直接写表。

### 2. 短信提醒先落第一版学生主动触达

- 第一版学生主动提醒先用短信打通，不等待小程序或公众号完全就绪。
- 短信只承接“提醒投递”，不承接业务状态。
- 第一版优先覆盖：
  - 开课前提醒
  - 临时调课 / 变更提醒
  - 关键待办截止提醒
  - 其他通道尚未接入时的高优先级兜底提醒
- 后续接入小程序、服务号或其他微信通道时，短信保留为兜底能力，而不是被视作临时脏方案。

### 3. 销售前端接入成为 Enrollment 之前的轻量入口

- 销售前端接入进入 V2，不再继续留作纯讨论项。
- 当前默认目标不是先做完整 CRM，而是先把“销售沟通 -> 学生档案草稿 -> Enrollment/录取接入”这条业务链打通。
- 第一版销售前端至少需要承接：
  - 跟进了哪个学生
  - 沟通了什么内容
  - 当前意向与风险
  - 下一步动作
  - 是否生成学生档案草稿
  - 是否推进到 Enrollment / 录取

### 4. 腾讯会议接入课后反馈

- 腾讯会议在 V2 中不作为单独会议工具项目处理，而是作为课后反馈的课堂证据来源。
- 目标不是把腾讯会议变成业务真相源，而是把录制、转写、纪要能力接入现有反馈流。
- 第一版默认链路是：
  - 课次结束
  - 拉取录制 / 转写 / 纪要
  - AI 生成“上课表现”草稿
  - 老师补充“学生反馈 / 主观判断 / 后续建议”
  - 老师审核后提交正式反馈

### 5. 内部外化入口继续保留，但不再是唯一先手

- `OpenClaw + Feishu` 仍然是 V2 的有效内部效率方向。
- 但它不再被视为当前唯一 Phase 1 主线。
- 更合理的位置是：
  - 复用提醒 / 动作底座
  - 作为内部 admin / teacher / manager 的后续站外入口
  - 在学生提醒、销售前端、腾讯会议反馈三条主线落稳后并行推进

### 6. 项目交付管理成为 V2 后续主产品层

- `Enrollment` 保留为“已确认服务”的入口对象，不直接承载完整项目交付状态。
- V2 后续仍默认引入：
  - `ProjectTrack`
  - `SubProject`
  - `Milestone`
  - `Risk Board`
- 这层模型负责回答“项目走到哪、接下来到期什么、谁负责、哪里要预警”，而不是只回答“上了哪节课”。

## V2 当前锁定的三条开发主线

### 1. 短信提醒接入

- 第一版优先服务学生侧提醒，不先拿短信做内部协同主入口。
- 短信的职责是：
  - 触达高优先级提醒
  - 在微信 / 小程序尚未接入前先保业务可执行
  - 为后续多通道提醒建立统一发送、失败重试、送达记录和审计规则
- 短信不负责：
  - 业务状态确认
  - 学生回复驱动的复杂流程
  - 替代网站中的正式处理动作

### 2. 销售前端接入

- 第一版定位为“轻量售前入口”，不是完整销售 CRM。
- 这条链路默认回答三个问题：
  - 我们和哪个学生 / 家长沟通过
  - 当前达成了什么共识、暴露了什么风险
  - 什么时候进入正式学生档案与 Enrollment 流程
- 第一版最重要的产物不是 pipeline 看板，而是：
  - 沟通记录
  - 学生档案草稿
  - 向 Enrollment 的可追踪 handoff

### 3. 腾讯会议接入课后反馈

- 腾讯会议接入的核心不是“会议管理”，而是“课堂内容证据自动进入反馈流程”。
- 第一版明确分工：
  - 腾讯会议提供录制 / 转写 / 纪要等课堂素材
  - AI 负责生成结构化“上课表现”草稿
  - 老师负责修正、补充学生反馈和主观判断
  - 网站中的 `CourseFeedback` 仍是最终反馈真相源
- 第一版不做“会议结束即自动提交正式反馈”。

## 共享底座与接口边界

### 1. 网站继续做唯一真相源

- 短信不是状态源。
- 腾讯会议不是状态源。
- 未来微信、小程序、OpenClaw、Feishu 都不是状态源。
- 正式的状态迁移、反馈提交、录取推进、学生档案落库都必须回到网站服务层完成。

### 2. 提醒能力用统一事件底座承接

- 待办提醒、课程提醒、临时变更提醒都应继续走统一事件出口。
- 统一管理：
  - 谁应该被提醒
  - 通过哪个通道提醒
  - 是否投递成功
  - 是否需要重试
  - 是否已经有审计记录
- 短信只是第一条先落的学生提醒通道，不应另起一套孤立逻辑。

### 3. 销售前端通过网站业务服务进入 Enrollment

- 销售前端应被视为 Enrollment 之前的前置入口，而不是新的真相源系统。
- 沟通记录、学生档案草稿、录取推进都应通过网站业务边界完成。
- 第一版不做自由扩张的售前对象 CRUD，也不直接引入重 CRM 模型。

### 4. 腾讯会议接入只负责生成反馈草稿

- 会议录制 / 转写 / 纪要进入网站后，最多生成“待审核反馈草稿”。
- 老师审核通过前，不进入正式课后反馈状态。
- AI 输出只是一段辅助内容，不应绕过老师成为正式评价。

## 当前阶段明确不做

- 不把短信做成双向客服主平台
- 不把销售前端第一版做成完整 CRM、商机池或复杂 pipeline 系统
- 不让腾讯会议录制结果自动提交正式反馈
- 不让短信、腾讯会议、OpenClaw、Feishu 成为业务主状态源
- 不把网站后台原样搬成外部通道后台

## V2 的当前默认业务模型

以下对象仍是 V2 后续主产品层默认方向，但不属于当前三条开发主线的已实现能力：

### 1. Enrollment 继续保留

- `Enrollment` 仍表示已确认合作 / 已确认服务。
- 它不直接变成项目交付对象。

### 2. ProjectTrack

- 表示一个学生的一条长期服务主线。
- 一个 `Enrollment` 默认对应一个 `ProjectTrack`。

### 3. SubProject

- 表示长期服务中的一个子项目单元。
- 用来承接 `2+1`、`1+1+1` 这种长期服务结构。
- 每个 `SubProject` 都有自己的负责人、阶段、节点、状态。

### 4. Milestone

- 表示交付节点，例如：
  - 技术评估完成
  - 项目计划书冻结
  - Demo 初版完成
  - 内部验收完成
  - 可授课
  - 商业部分开始
  - 拍摄完成
  - 比赛报名提交
  - 结课
  - 文书 / 进程文档 / 代码交付

### 5. Risk Board

- 用于把未来 3/5/7 天和已逾期节点汇总出来。
- 同时承接“提醒要发给谁、通过什么渠道发”的管理视角。

## V2 的默认流程原则

- 优先把真实业务链打通，不先做重平台壳。
- 网站业务层继续拥有状态，外部渠道只消费事件、回调动作、素材或跳转详情。
- 销售前端先解决“沟通沉淀 -> 档案草稿 -> Enrollment handoff”，不先追求完整售前平台化。
- 课后反馈先解决“自动预填 + 老师审核”，不先追求完全自动化。
- 学生提醒先用低摩擦、可落地的短信通道，不等待理想化全量微信体系。
- 内部 OpenClaw / Feishu 继续保留，但优先级服从当前三条开发主线。

## V2 的功能优先级重排

### P0：短信提醒接入

- 学生高优先级提醒模板
- 发送记录、失败重试、审计记录
- 课程 / 待办 / 临时变更类提醒接入

### P1：销售前端接入

- 沟通记录
- 学生档案草稿
- 向 Enrollment / 录取的 handoff

### P2：腾讯会议接入课后反馈

- 会后录制 / 转写拉取
- AI 生成上课表现草稿
- 老师审核后写入正式反馈

### P3：内部站外入口

- `OpenClaw + Feishu`
- 角色摘要读接口
- 动作白名单与安全写网关

### P4：项目交付管理层

- `ProjectTrack / SubProject / Milestone / Risk Board`
- 模板化项目类型

### P5：后续学生多通道能力

- 小程序 / 服务号等微信通道
- 学生提醒偏好
- 多通道路由和兜底策略

## Open Decisions / Not Yet Locked

以下内容仍未锁死，后续规划或实现前必须先确认：

- 短信供应商最终选型与模板策略
- 短信接入后，小程序 / 服务号在 V2 中何时进入实施
- 销售前端第一版到底包含哪些最小字段和最小角色边界
- 学生档案草稿与正式 `Enrollment` 的切换时点
- 腾讯会议接入的鉴权方式、企业版本要求和回调策略
- AI 反馈草稿的模板结构与老师审核规则
- `OpenClaw + Feishu` 进入并行开发的具体时点

## V2 与现有系统怎么衔接

- 和 `Enrollment` 关联，但不覆盖 `Enrollment` 的现有语义。
- 和 `CourseSchedule` 联动，用于驱动上课提醒和课后反馈生成。
- 和 `CourseFeedback` 联动，让腾讯会议转写进入反馈草稿链路。
- 和现有提醒 / 审计底座联动，承接短信投递和后续多通道扩展。
- 和未来 `OpenClaw / Feishu / 微信` 通道联动，但这些通道都不成为真相源。

## V2 先不做什么

- 不把完整售前 CRM 和销售 pipeline 一次性并入第一版实现。
- 不把短信当作长期唯一学生通道。
- 不做完全无人审核的 AI 反馈自动提交。
- 不让腾讯会议、短信、OpenClaw、Feishu 成为业务主状态源。
- 不在第一阶段就做复杂甘特图和重 BI。

## V2 成功标准

- 网站仍然是唯一业务母端口和对外接口源。
- 学生高优先级提醒可以通过短信稳定触达，并留下投递记录。
- 销售沟通内容能够沉淀为学生档案草稿，并顺滑进入 Enrollment / 录取流程。
- 腾讯会议的课堂素材能够生成课后反馈草稿，老师只需要补充和审核。
- 后续内部外化入口和项目交付模型，可以建立在同一套底座之上，而不是另起真相源。
