# V2 目标：交付管理与外化能力

更新时间：2026-03-23

## 一句话定义

V2 的首要任务不是继续把更多动作锁在网站里，而是在保持网站业务层与 workflow-safe API 为唯一真相源的前提下，先完成 `OpenClaw + Feishu` 的内部外化提醒与动作网关，再在此基础上推进项目交付管理层。

## V2 现在要解决什么

V1 已经具备站内执行闭环，但当前仍有三个直接缺口：

- 不进网站，就看不到待办、风险和关键动作
- 当前 `/oa/api/external/*` 已有雏形，但更像对象 CRUD，不是安全动作 API
- 系统追踪的重点仍是课次执行，不是项目交付 readiness

这会带来直接问题：

- 教务、老师、老板仍要回网站才能处理高频动作
- 站外通道还没有统一入口，OpenClaw、Feishu、未来学生提醒难以复用
- 现有 external 写口如果直接扩张，容易绕过网站业务层和 workflow 约束
- 项目交付相关的里程碑、风险、节点预警仍未进入系统一等对象

## V2 当前默认目标顺序

### 1. 保住网站母端口

- 网站业务层和数据库继续是单一事实源。
- 当前 `auth + enrollment + workflow + oa + feedback + chat` 接口语义要保留。
- OpenClaw、Feishu、微信、未来其他渠道都只能通过网站业务层或 workflow-safe API 接入，不能直接写表。

### 2. 收口现有 external 风险写口

- 当前 `/oa/api/external/*` 已证明外化方向成立，但其中部分写口仍偏薄。
- V2 Phase 1 开始前，要先把高风险 external 写口收口成服务层动作，而不是继续扩展通用 CRUD。
- 当前重点风险包括：
  - `leave approve/reject` 过薄
  - `schedule` external 直接 CRUD
  - `todo` external 直接 CRUD，且不能继续允许碰 workflow todo

### 3. 建立提醒与动作外化底座

- 为待办提醒、风险提醒、上课提醒、确认提醒建立统一事件出口。
- 至少需要一层事件记录、投递记录、用户映射与审计能力。
- 通知只作为事件消费者，不拥有业务状态。

### 4. 内部提醒与关键动作先站外化

- 第一优先级是让内部团队先在站外看到摘要并执行关键动作，而不是继续要求大家先登录网站。
- 当前已锁定的第一阶段方向是 `OpenClaw + Feishu`。
- 网站继续承接详情页、审批页、上下文页和最终状态迁移。

### 5. 项目交付管理成为 V2 后续主产品层

- `Enrollment` 保留为“已确认服务”的入口对象，不直接承载里程碑状态。
- V2 后续仍默认引入：
  - `ProjectTrack`
  - `SubProject`
  - `Milestone`
  - `Risk Board`
- 这层模型负责回答“项目走到哪、接下来到期什么、谁负责、哪里要预警”，而不是只回答“上了哪节课”。

### 6. 外部学生提醒能力

- 外部学生提醒仍属于 V2 范围，但不进入第一阶段实现。
- 第一阶段不做学生站外入口；学生提醒和学生站外动作留到下一阶段决定通道后再落。

### 7. 销售前端接入

- 销售前端接入仍是重要方向，但当前不锁定进入第一阶段。
- 当前更合理的写法是：
  - 先作为待确认是否提前拉入 V2 的轻量入口
  - 完整销售 / 售前平台化仍按 V3 默认方向保留

## V2 Phase 1 已锁定方案

- 第一阶段只做内部用户，不做学生外部入口。
- 第一阶段采用 `OpenClaw + Feishu`。
- 查询走角色摘要接口，不镜像整页 dashboard 或 action center。
- 修改走 OpenClaw 专用动作网关，不做全量 API 后门。
- 鉴权采用“服务 Token + 用户映射 + 服务端二次权限校验”。
- 请求方传入的 `user=admin`、`user=liyu` 这类自报身份不作为可信身份。
- 网站业务层和 workflow-safe API 继续做唯一状态迁移入口。
- 第一阶段的“尽量全覆盖”解释为“内部高频安全动作全覆盖”，不是“后台级原始 CRUD 全开放”。

## 第一阶段接口边界

### 读接口

- 角色摘要接口优先，不复刻整页 payload。
- 当前默认会有：
  - `me/summary`
  - `me/schedules`
  - `me/work-items`
  - 对象详情接口，例如 enrollment / workflow / schedule 详情
  - reminder 列表接口
- 这些接口面向 OpenClaw 和飞书卡片消费，返回的是站外摘要与可执行上下文，不是网站页面镜像。

### 写接口

- 第一阶段统一采用 `command` 动作网关。
- OpenClaw 请求只表达：
  - 谁发起
  - 想做什么动作
  - 携带什么业务参数
- 服务端负责：
  - 校验服务 Token
  - 完成外部身份到站内用户映射
  - 按站内角色、对象归属和 workflow 状态二次校验
  - 调用网站业务服务或 workflow-safe API 完成动作

### 提醒与审计对象

- 第一阶段需要的规划对象包括：
  - `ExternalIdentity`
  - `ReminderEvent`
  - `ReminderDelivery`
  - `IntegrationActionLog`
- 这些对象属于提醒与集成底座，不代表 Feishu 或 OpenClaw 成为业务状态源。

## 第一阶段动作白名单

### 教务

- 自动匹配排课
- 手动排课方案保存
- 发送方案给学生
- 请假审批 / 驳回
- 安全调课动作
- 账号创建
- 账号更新
- 账号停用 / 启用
- 重置密码

### 老师

- 提案预检
- 提案提交
- 反馈保存
- 反馈提交

### 老板 / 管理层

- 只读摘要
- 只读风险面板
- 不提供第一阶段写动作

## 第一阶段明确不做

- 不做学生外部入口
- 不做 chat 外化
- 不做全量 `schedule CRUD` 透传
- 不做全量 `todo CRUD` 透传
- 不做 `user hard delete`
- 不做“共享管理员身份”
- 不做把现有站内 API 原样搬成飞书后门

## V2 的当前默认业务模型

以下对象仍是 V2 后续主产品层默认方向，但不属于 Phase 1 已实现能力：

### 1. Enrollment 继续保留

- `Enrollment` 仍表示已确认合作 / 已确认服务。
- 它不直接变成项目交付对象。

### 2. ProjectTrack

- 表示一个学生的一条长期服务主线。
- 一个 `Enrollment` 默认对应一个 `ProjectTrack`。

### 3. SubProject

- 表示长期服务中的一个子项目单元。
- 用来承接 `2+1`、`1+1+1` 这种长期服务结构。
- 每个 `SubProject` 都有自己的负责人、阶段、节点、状态。

### 4. Milestone

- 表示交付节点，例如：
  - 技术评估完成
  - 项目计划书冻结
  - Demo 初版完成
  - 内部验收完成
  - 可授课
  - 商业部分开始
  - 拍摄完成
  - 比赛报名提交
  - 结课
  - 文书 / 进程文档 / 代码交付

### 5. Risk Board

- 用于把未来 3/5/7 天和已逾期节点汇总出来。
- 同时承接“提醒要发给谁、通过什么渠道发”的管理视角。

## V2 的默认流程原则

- 优先做风险预警和提醒外化，不先做强阻断卡控。
- 网站业务层继续拥有状态，外部渠道只消费事件、回调动作或跳转详情。
- 复用现有 `OATodo` 承接执行任务，但不要让 `OATodo` 直接替代项目模型。
- 内部优先，先解决 SCF 团队日常执行效率，再考虑完整对外产品化包装。
- 设计外化能力时，优先建设安全动作边界，而不是先扩张通用对象 CRUD。

## 默认支持的项目类型

### 纯技术项目

- 技术评估
- 计划书确认
- Demo 初版
- 内部验收
- 技术授课
- 结课输出

### 技术 + 商业项目

在纯技术项目基础上增加：

- 商业部分启动
- 商业授课
- 拍摄准备 / 拍摄完成

### 技术 + 商业 + 比赛项目

在技术 + 商业项目基础上增加：

- 比赛材料准备
- 比赛报名提交
- 赛后归档

## V2 默认预警 / 提醒视图

- 未来 5 天关键节点
- 未来 7 天关键节点
- 已逾期节点
- 已临近但前置条件未完成的节点
- 我的待办提醒
- 我的风险提醒
- 第一阶段内部角色摘要

## V2 的功能优先级重排

### P0：收口现有 external 风险写口

- 收口 `leave approve/reject`
- 收口 `schedule` external 风险写口
- 收口 `todo` external 风险写口
- 禁止 workflow todo 被通用 external todo 接口继续修改

### P1：网站母端口与接口保留

- 保住现有网站接口语义
- 为外部通道准备复用边界

### P2：提醒 / 通知外化底座

- 用户映射
- 事件出口
- 投递记录
- 审计记录

### P3：OpenClaw 动作网关与内部飞书入口

- 角色摘要读接口
- `command` 写网关
- 教务 / 老师 / 老板的飞书入口

### P4：项目交付管理层

- `ProjectTrack / SubProject / Milestone / Risk Board`
- 模板化项目类型

### P5：外部学生提醒

- 学生通道
- 学生提醒
- 学生站外动作

### P6：销售前端接入

- 仍待 leadership 决策
- 若提前进入，只拉轻量入口，不一次性做完整售前平台

## Open Decisions / Not Yet Locked

以下内容仍未锁死，后续规划或实现前必须先确认：

- 外部学生提醒第一版具体走什么通道
- 销售前端接入到底属于 V2 的轻量前置能力，还是继续留在 V3
- 数据模型最终形态是否为完整 `ProjectTrack / SubProject / Milestone` 三层，还是先落更轻的第一版
- 风险面板默认按学生、项目、负责人还是时间窗口聚合

这些未决项也同步记录在 `docs/handoff/current-focus.md` 中；如果两边出现冲突，以最新更新的 handoff 和后续确认结果为准。

## V2 与现有系统怎么衔接

- 和 `Enrollment` 关联，但不覆盖 `Enrollment` 的现有语义。
- 和 `CourseSchedule` 联动，用于识别“还没 ready 但课程已经临近”。
- 和 `OATodo` 联动，把节点拆成认领任务和执行待办。
- 和网站业务服务联动，把站内动作外化到 OpenClaw / Feishu / 未来学生提醒通道。
- 和 `handbook` / 现有输出能力联动，承接结课输出物。

## V2 先不做什么

- 不把完整售前 CRM 和销售 pipeline 一次性并进第一阶段实现。
- 不让 Feishu、OpenClaw、微信成为业务主状态源。
- 不在第一阶段就做复杂甘特图和重 BI。
- 不默认做自由双向同步的外部渠道集成。
- 不把网站后台直接搬成飞书后台。

## V2 成功标准

- 网站仍然是唯一业务母端口和对外接口源。
- 内部团队不进网站，也能先收到结构化提醒并执行关键动作。
- 第一阶段的站外动作都通过网站业务层和 workflow-safe API 完成。
- 老板 / Admin 能直接看到未来 3/5/7 天风险摘要。
- 后续项目交付模型可以建立在同一套提醒与动作底座之上，而不是另起真相源。
