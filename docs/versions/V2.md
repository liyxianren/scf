# V2 目标：项目交付管理

更新时间：2026-03-21

## V2 要解决什么

V2 不再只是继续加 OA 待办，而是引入一层独立的“项目交付管理”，用来解决以下问题：

- 当前项目到底走到哪个阶段
- 关键交付节点什么时候到
- 谁负责，谁认领，谁还没推进
- 哪些项目未来 3/5/7 天内会有风险

以下内容是当前默认方向，不代表所有实现决策都已经锁死。

## 为什么 V2 必须现在做

当前系统已经能管理报名、排课、请假、补课、课后反馈，但它追踪的是“课次执行”，不是“项目交付 readiness”。

这会导致典型问题：

- 明天要拍摄，但今天 Demo 还没 ready
- 要开始商业部分，但老师不知道前置交付没有完成
- 比赛报名临近，但材料节点没有被系统提醒

## V2 的当前默认业务模型

### 1. Enrollment 继续保留

- `Enrollment` 仍表示已确认合作 / 已确认服务。
- 它不直接变成项目交付对象。

### 2. ProjectTrack

- 表示一个学生的一条长期服务主线。
- 一个 `Enrollment` 默认对应一个 `ProjectTrack`。

### 3. SubProject

- 表示长期服务中的一个子项目单元。
- 用来承接 `2+1`、`1+1+1` 这种长期服务结构。
- 每个 `SubProject` 都有自己的负责人、阶段、节点、状态。

### 4. Milestone

- 表示交付节点，例如：
  - 技术评估完成
  - 项目计划书冻结
  - Demo 初版完成
  - 内部验收完成
  - 可授课
  - 商业部分开始
  - 拍摄完成
  - 比赛报名提交
  - 结课
  - 文书 / 进程文档 / 代码交付

### 5. Risk Board

- 用于把未来 3/5/7 天和已逾期节点汇总出来。
- 这是 V2 最关键的管理视图之一。

## V2 的默认流程原则

- 不做强阻断式卡控作为第一步，先做风险预警。
- 优先让 Admin、老板、执行人看到节点和风险，而不是依赖人脑记忆。
- 复用现有 `OATodo` 承接执行任务，但不要再让 `OATodo` 直接替代项目模型。
- 保持当前 Flask 单体与数据库为主事实源，不引入外部系统为主状态源。

## 默认支持的项目类型

### 纯技术项目

- 技术评估
- 计划书确认
- Demo 初版
- 内部验收
- 技术授课
- 结课输出

### 技术 + 商业项目

在纯技术项目基础上增加：

- 商业部分启动
- 商业授课
- 拍摄准备 / 拍摄完成

### 技术 + 商业 + 比赛项目

在技术 + 商业项目基础上增加：

- 比赛材料准备
- 比赛报名提交
- 赛后归档

## V2 默认预警视图

- 未来 3 天关键节点
- 未来 5 天关键节点
- 未来 7 天关键节点
- 已逾期节点
- 已临近但前置条件未完成的节点

## Open Decisions / Not Yet Locked

以下内容仍未锁死，后续规划或实现前必须先确认：

- 数据模型最终形态是否为单独的 `ProjectTrack / SubProject / Milestone` 三层，还是需要更轻的第一版落地方式
- API 入口与页面入口应该先落在现有 OA 内，还是拆出独立项目管理入口
- 里程碑模板是否支持批量生成，以及模板与项目类型的绑定方式
- 风险面板默认按学生、项目、负责人还是时间窗口聚合
- `CourseSchedule` 与 `OATodo` 的联动深度，包括哪些事件需要自动生成任务或预警

这些未决项也同步记录在 `docs/handoff/current-focus.md` 中；如果两边出现冲突，以最新更新的 handoff 和后续确认结果为准。

## V2 与现有系统怎么衔接

- 和 `Enrollment` 关联，但不覆盖 `Enrollment` 的现有语义。
- 和 `CourseSchedule` 联动，用于识别“还没 ready 但课程已经临近”。
- 和 `OATodo` 联动，把节点拆成认领任务和执行待办。
- 和 `handbook` / 现有输出能力联动，承接结课输出物。

## V2 先不做什么

- 不把售前学生档案 CRM 并进第一阶段实现。
- 不在第一阶段就做复杂甘特图。
- 不把外部集成作为主依赖。

## V2 成功标准

- 一个 `Enrollment` 下可挂多个 `SubProject`。
- 每个子项目有清晰阶段和节点。
- 老板 / Admin 能直接看到未来 3/5/7 天风险。
- 交付风险不再依赖人脑记忆。
