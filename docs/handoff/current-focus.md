# 当前接手焦点

更新时间：2026-03-24
最近一次工作区校验：`git status --short` on 2026-03-24

## 当前真实情况

- 当前仓库已经具备 `auth + enrollment + workflow + oa + feedback + chat` 的内部交付闭环雏形。
- 网站端继续是当前业务信息的母端口，后续对内对外通道都应复用网站业务层与接口边界。
- V2 当前活跃主线已改为三条并行开发流：
  - 短信提醒接入
  - 销售前端接入
  - 腾讯会议接入课后反馈
- `OpenClaw + Feishu` 仍然保留为 V2 的内部外化方向，但不再是当前唯一先手。
- 项目交付管理仍未建模，继续留在 V2 后续主产品层，不压过当前三条开发主线。

## 当前工作流重点

### 1. OA P1 稳定化仍需保留

- 当前已存在的 OA / workflow 闭环仍然是后续所有集成的落点，不要因为新主线而绕开现有业务边界。
- 当前这轮已完成的 OA P1 重点包括：
  - 三端 action center 已形成 `下一步动作` 聚合视图
  - 请假驳回说明、补课偏好、补课确认回写、提案 warning 已进入结构化 payload
  - 学生确认中心可预览 workflow 方案
  - 学生首页已展示完整老师反馈
- 代码与测试事实：
  - `pytest -q tests/integration` on 2026-03-23 -> `67 passed`

### 2. V2 当前第一主线：短信提醒接入

- 第一版学生主动提醒先用短信落地，不等待小程序或公众号完全就绪。
- 短信的职责是高优先级提醒与兜底提醒，不承接业务状态。
- 当前需要优先明确：
  - 供应商选型
  - 提醒模板类型
  - 发送记录、失败重试和审计边界
  - 与现有 reminder/event 底座的复用方式

### 3. V2 当前第二主线：销售前端接入

- 当前默认方向已从“是否进入 V2”变为“进入 V2，但只做轻量前置入口”。
- 第一版销售前端的核心不是完整 CRM，而是打通：
  - 沟通了哪个学生
  - 沟通了什么内容
  - 当前下一步是什么
  - 如何生成学生档案草稿
  - 如何推进到 Enrollment / 录取

### 4. V2 当前第三主线：腾讯会议接入课后反馈

- 腾讯会议接入不单做会议工具联动，而是直接服务课后反馈自动预填。
- 当前默认链路是：
  - 课次结束后拉会议录制 / 转写 / 纪要
  - AI 生成“上课表现”草稿
  - 老师补充“学生反馈 / 主观判断 / 后续建议”
  - 老师审核后提交正式反馈
- 老师审核前，会议素材和 AI 输出都不能直接变成正式反馈状态。

### 5. OpenClaw / Feishu 进入次主线

- 已有的外化提醒与动作网关思路仍然有效。
- 但当前不再把 `OpenClaw + Feishu` 视为唯一 Phase 1。
- 更合理的定位是：
  - 复用提醒与动作底座
  - 等当前三条主线落稳后，再作为内部站外入口继续推进

## 推荐的下一步

1. 先锁短信供应商、模板范围和送达记录结构。
2. 明确销售前端第一版的最小字段、角色范围和转 Enrollment 边界。
3. 明确腾讯会议的鉴权方式、录制/转写拉取方式和反馈模板映射。
4. 在这三条主线稳定后，再决定 `OpenClaw + Feishu` 的并行落地时点。

## 当前阻塞与开放点

- 短信供应商最终采用哪家、模板审核与签名策略怎么落，仍未锁死。
- 小程序 / 服务号什么时候从“后续通道”进入实施，仍未锁死。
- 销售前端第一版是否引入独立线索对象，还是只做 Enrollment 前置草稿，仍未锁死。
- 学生档案草稿和正式 Enrollment 的切换时点，仍未锁死。
- 腾讯会议企业版能力、录制转写可用性和回调策略，仍需在实现前再校准。
- AI 反馈草稿的字段模板、老师审核规则和最终 UI 承载方式，仍未锁死。
