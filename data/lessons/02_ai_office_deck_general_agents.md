## 通用 Agent 与 Openclaw 5 层

上一节我们写了一个 50 行的 Agent 骨架。这一节要回答的是：

- **为什么我们需要"通用 Agent 框架"？** 自己写不行吗？
- **通用 Agent 多出来的能力是什么？** Skills、子 Agent、MCP 分别解决什么问题？
- **Openclaw 是怎么设计的？** 它的 5 层架构每一层在解决什么问题？（本节重点）
- **Openclaw / Codex / Claude Code 怎么选？** 三家定位差别在哪？

这一节是整个培训中**信息密度最高的一节**。我们会从工程落地的视角，把"Agent 框架"这件抽象的事拆到看得见、摸得着。

## 手写 50 行到生产级：要踩的 5 个坑

上一节我们写了一个 50 行能跑的 Agent，但如果你把它推到公司里给整个团队用，马上会撞上五个经典问题：

- **多用户并发**：5 个人同时用，他们的对话历史会不会串？一个人跑到一半别人的指令进来怎么办？
- **模型切换**：今天用 Claude，明天想试 GPT-5，再后天想上本地 Ollama——总不能每次都重写业务代码吧？
- **长任务记忆**：跑了 2 小时的任务，上下文早就溢出了。Agent 还记得最初的任务要求吗？
- **工具失败**：某个工具返回了乱码或超时，能不能不让整个任务崩掉？
- **审计追溯**：两周前某次执行跑飞了，能不能精确回放当时发生了什么？

一个人写到第 3 条就会开始怀疑人生。所以就有了**通用 Agent 框架**。

> 通用 Agent 框架做的事，就是把这 5 个工程问题一次性打包解决，让你只专注业务。

## 通用 Agent 的三大特性

<img src="/static/images/ai-office/skills-subagents.svg" alt="Skills 与子 Agent" />

通用 Agent 框架比"手写循环"多出来的能力，主要是三个：

- **Skills（技能）**——可复用的领域知识包。按触发词自动挂载到系统提示
- **Sub-agents（子 Agent）**——派出去处理子任务的独立执行体，回来只交结论
- **MCP / Tool 生态**——标准化的工具对接协议，跨框架复用

这三个特性把 Agent 从"一个循环"升级成了"可以组合、可以扩展、可以治理"的系统。下面我们一个一个讲。

---

## 特性一：Skills 按需挂载

**Skill 是一个可复用的领域知识包**——通常是一个目录，里面放一份 `SKILL.md`（说明书）和一些辅助脚本或模板。

`SKILL.md` 告诉 Agent 三件事：

- **什么时候该用这个 Skill**——触发词 / 触发条件
- **这个 Skill 里装着什么步骤和约定**
- **可以调用哪些工具、按什么格式产出**

当你在对话里提到相关关键词，框架会**自动把对应 Skill 的内容并入当前系统提示**。Agent 拿到额外知识，就能按这个领域的范式工作。

本仓库里就有真实的 Skills：

- `skills/pptx/` — 做 PPT 的 Skill
- `skills/docx/` — 做 Word 的 Skill
- `skills/xlsx/` — 做 Excel 的 Skill
- `skills/pdf/` — 做 PDF 的 Skill
- `skills/scf-platform-context/` — SCF 项目的业务 framing Skill

> Skills 把"领域 know-how"从你每次的 Prompt 里抽出来，写一次，全员受益。

---

## 特性二：Sub-agents 派出去的专项小组

**子 Agent 是主 Agent 派出去处理子任务的独立执行体。关键词是"独立"**——它有自己的上下文，做完只把结论回传给主 Agent。

打个比方：主 Agent 是项目经理，子 Agent 是派去做调研的下属。下属读了 100 份文件，最后只给经理一份 300 字的摘要报告——经理不用看那 100 份文件。

子 Agent 的三个核心价值：

- **保护主上下文**：100 份文件都进主 Agent 会撑爆上下文，子 Agent 只回传摘要就干净了
- **并行加速**：可以同时派 3 个子 Agent 做独立调研，比串行快很多
- **能力分离**：不同子 Agent 可以配不同工具权限、不同模型

Claude Code 内置的典型子 Agent：

- `Explore` — 专门做探索式代码搜索，读一堆文件只回结构化摘要
- `Plan` — 专门做设计实现方案，不改代码，只产出计划

> Skills 改变"主 Agent 怎么想"，子 Agent 改变"谁来做这件事"——两者经常一起用。

---

## 特性三：MCP / Tool 生态

**MCP（Model Context Protocol）** 是 Anthropic 主导的一套开放协议，解决的问题是：

> 每个外部系统（Jira、Slack、Gmail、数据库）都要给 Agent 开个口子——怎么把这套对接标准化？

过去每家 Agent 框架都自己写一套工具对接层，换个框架就得全部重写。MCP 把这件事标准化了：

- 一个外部系统只需要实现一个 **MCP Server**
- 任何支持 MCP 的 Agent 框架都能直接接入

结果：**你为 Jira 写一次 MCP Server，Claude Code、Codex、未来的其他框架都能用**。

这和 USB 统一所有硬件接口是一个意思——让 Agent 和工具之间变成可插拔的关系。

> MCP 是 Agent 生态走向开放协作的关键一步。

## Openclaw 5 层架构总览

<img src="/static/images/ai-office/openclaw-5layers.svg" alt="Openclaw 5 层架构" />

Openclaw 的工程亮点在于把 Agent 拆成了 **5 个清晰的层**，每一层职责单一，可独立替换、独立升级：

- **Layer 1 · 接口与输入层**（感官）——多渠道接入，任务的起点
- **Layer 2 · 网关控制平面**（交通枢纽）——标准化 + 隔离 + 并发控制
- **Layer 3 · Agent 运行器**（大脑）——理解、拆解、规划、上下文管理
- **Layer 4 · 执行与工具层**（双手）——把决策变成真实操作
- **Layer 5 · 混合内存系统**（长期记忆）——解决 Agent 的"失忆"问题

下面我们逐层过一遍，重点在 Layer 2、Layer 3、Layer 5——这三层是工程亮点最密集的地方。

> 看到这一页你要记住一个原则：**每一层只做一件事，层与层之间通过标准接口连接**。

---

## Layer 1：接口与输入层

这是 Agent 的感官，所有任务的起点。它的核心作用是**实现多渠道接入**：

- **CLI（终端命令行）**——最核心的输入方式，面向开发者和内部运营
- **WhatsApp / Telegram**——个人端即时通讯，面向终端用户
- **Discord / Slack**——团队协作场景
- **Cron（定时任务）**——每天早上 8 点自动跑一次这类无人值守触发
- **Webhooks（回调）**——由外部事件触发，比如 GitHub 有 PR 提交就跑

这一层的设计要点是：**每个渠道都是独立的适配器，互不影响**。新接入一个渠道（比如企业微信），只要加一个适配器，**下游任何一层都不用动**。

> 接入层的解耦度，决定了 Agent 能不能快速扩展到新场景。

---

## Layer 2：网关控制平面

这是 Openclaw 最核心的工程亮点之一。它的作用是**标准化、隔离、并发控制**，包含三大组件：

**通道适配器**——负责输入的归一化：
- 不同渠道来的消息格式不同（WhatsApp vs CLI vs Webhook JSON）
- 适配器把它们统一转换成标准结构
- 下游完全不需要关心消息从哪来的

**会话路由器**——负责会话隔离：
- 为每个用户、每个任务创建独立会话
- 根据 user_id / task_id 路由到对应会话
- **彻底避免跨会话状态污染和上下文串扰**——用户 A 的历史不会跑到用户 B 的推理里

**车道队列**——负责并发控制（下一页专门讲）。

> 这一层解决的是：**多人同时用一个 Agent 平台时，怎么让每个人看到的是"只属于自己的 Agent"**。

---

## Layer 2 精讲：车道队列

<img src="/static/images/ai-office/layer2-lanes.svg" alt="车道队列" />

**核心设计**：每个会话分配一条专属车道，车道内事件严格按 FIFO 串行执行。

考虑一个场景——用户 A 连发 3 条消息：**"查天气"、"发邮件"、"写总结"**。

如果这 3 条并发进入同一个 Agent，会发生灾难：

- 它们可能同时修改对话历史
- 可能同时读写同一块内存
- 可能同时调用同一个外部工具
- 执行顺序不确定，结果不可复现

车道队列的答案是：**同一会话内永远只有一个事件在跑，严格 FIFO**。

- ✅ 没有竞态条件
- ✅ 没有状态撕裂
- ✅ 执行顺序可预测、可复现

而**不同车道之间完全并行**——用户 A 的车道不会阻塞用户 B 的。整体系统照样高并发。

> 类比：银行柜台。每个柜台一条队（串行），所有柜台同时开（并行）。串行和并行在不同粒度上兼得。

---

## Layer 3：Agent 运行器

这是 Agent 的大脑，负责理解任务、拆解规划、管理上下文。它由三个组件组成：

**模型解析器**——屏蔽具体模型：
- 支持 Claude、OpenAI、本地部署的 Ollama 等主流模型
- 根据任务特点动态选型：简单文件处理用本地轻量模型（省钱），复杂任务规划用 Claude Opus（能力强）
- 业务代码完全不感知用的是哪个模型

**系统提示词构建器**——把散落的信息拼装成高质量 system prompt：
- 任务核心要求
- 工具能力说明
- 内存里的历史信息
- 用户的个性化规则（Skills 里的约定）

**上下文窗口防护**——下一页专门讲，这是 Openclaw 的王牌设计。

> 这一层决定了 Agent 发挥稳不稳。模型选错、提示词拼得乱，都会直接体现在产出质量上。

---

## Layer 3 精讲：上下文窗口防护

<img src="/static/images/ai-office/layer3-context-guard.svg" alt="上下文防护" />

**这是解决 Agent 长周期任务失败的关键设计**。

多数粗糙的 Agent 实现：对话历史一旦超过上下文窗口，**从头开始静默截断**。结果是：

- **系统指令先被删**
- **任务要求跟着被删**
- Agent 拿到的是残缺的历史，开始胡乱发挥
- 最可怕的是——**你不知道它什么时候开始跑飞的**

Openclaw 的做法完全相反：

- **提前计算 token 占用**，不等溢出才处理
- 按**优先级**保留：
  - 必保留：系统指令、任务要求、最新结果、当前工具输出
  - 压缩：早期、中期对话 → 摘要成 1-2 段关键结论
  - 丢弃：冗长的原始工具输出 → 只留有用字段
- **可控降级，有完整日志**——你永远知道发生了什么

> 这一层让长周期任务（几小时甚至一整天）不会因为上下文溢出而无声跑飞。

---

## Layer 4：执行与工具层

这是 Agent 的双手，负责把大脑的决策转化为真实操作。两大模块：

**LLM API 调用模块**——统一封装不同大模型的调用接口：
- 流式响应（逐 token 吐出来，体验不卡）
- 错误重试（网络抖动不让任务挂）
- 成本统计（每次调用的 token 和花费都记下来）
- 异常兜底（模型挂了 / 限流 / 鉴权失败都有明确处理）

**沙箱运行时**——工具执行的核心载体：
- **Shell 命令执行**（Bash / Zsh）——代码运行、环境配置、文件批处理
- **无头浏览器**（Puppeteer）——网页操作、数据爬取、页面交互
- **文件系统访问**——文档处理、代码修改、数据归档

> 这三样加起来，几乎覆盖了办公自动化的所有能力——写代码、跑脚本、上网查、改文档、做报表。

---

## Layer 5：混合内存系统

<img src="/static/images/ai-office/layer5-memory.svg" alt="混合内存三层" />

这一层解决 Agent 最常见的"失忆"问题，是长周期任务的基础。**不同性质的记忆用不同存储**：

**JSONL 日志（原始记录）**
- 形态：每行一条 JSON 事件，append-only 持久化
- 内容：用户输入、模型规划、工具调用、执行结果、异常信息、token 统计
- 用途：审计追溯、故障回放、做后续摘要/索引的原料

**MEMORY.md（核心规则与知识）**
- 形态：结构化 Markdown，按主题分块
- 内容：用户偏好、团队规则、长期领域知识
- 用途：每一轮自动注入系统提示，跨会话、跨任务持久

**向量 / FTS5 索引（精准语义检索）**
- 形态：向量嵌入 + 全文索引
- 索引来源：历史日志、用户文档、内部知识库
- 用途：按当前任务"按需召回"，不走满上下文

> 三层协同：**日志管"发生过什么"，MEMORY.md 管"该怎么做"，向量管"去哪找细节"**。

## 三家横向对比 + 选型

<img src="/static/images/ai-office/agents-compare.svg" alt="三家对比" />

Openclaw、Codex、Claude Code 都是通用 Agent，但定位和强项完全不同：

| 维度 | **Openclaw** | **Codex**（OpenAI） | **Claude Code**（Anthropic） |
|---|---|---|---|
| 定位 | 企业级 Agent 平台 | 开发者代码/终端 Agent | 研发全流程 Agent |
| 核心强项 | 5 层架构 · 会话隔离 · 混合内存 | OpenAI 生态 · GPT 模型 | Skills · 子 Agent · MCP |
| 典型场景 | 多用户企业内部 Agent 中台 | 代码生成 · 终端任务 | 研发辅助 · 文档 · 数据处理 |
| 可控性 | 高（工程化导向） | 中 | 中 |
| 多用户并发 | 原生支持 | 需自解决 | 需自解决 |
| 模型可换 | 强 | 锁定 OpenAI | 锁定 Anthropic |

**选型一句话**：

- 多用户、多团队、要审计和治理 → **Openclaw**
- 纯个人、深绑 OpenAI 生态 → **Codex**
- 小团队、要丰富 Skills 生态 → **Claude Code**

> 不是"谁更好"，是"谁更适合你的场景"——就像货车、轿车、SUV，场景选对了才是好车。

## 小结 + 过渡到第 3 节

这一节的四个核心观点：

- **通用 Agent 的三大特性**：Skills / 子 Agent / MCP，分别解决知识复用、上下文保护、工具生态
- **Openclaw 5 层架构**：每层职责单一、可替换、可升级
- **最精妙的三个点**：Layer 2 的车道队列、Layer 3 的上下文防护、Layer 5 的混合内存
- **三家选型不是比强弱**：是比"定位是否匹配你的场景"

到这一步，我们已经讲完了"Agent 是什么"和"通用 Agent 框架是什么"。

下一节，我们从原理层跳到**实操层**——讲**怎么用 Prompt 指挥这些 Agent**，让它们真的帮你做 PPT、Word、Excel。

> 下一节是含金量最高的一节——你会看到同一个任务，Prompt 写得好和写得差，产出质量能差十倍。
