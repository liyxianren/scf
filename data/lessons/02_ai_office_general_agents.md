## 1. 从手写循环到通用 Agent

第 1 节我们写过一个 50 行的 Agent 循环骨架，它能跑通一个"查天气 + 发邮件"的 demo。但如果你真的在公司里用它做生产级应用，会踩到一堆坑：

- **多用户并发**：5 个人同时用，他们的上下文会不会串？一个人任务跑到一半，另一个人的指令突然进来了怎么办？
- **模型切换**：今天用 Claude，明天想试 GPT，再后天想上本地 Ollama，怎么不把所有业务代码重写一遍？
- **长任务记忆**：跑了 2 小时的任务，上下文早就溢出了，Agent 怎么"记住"前面发生过什么？
- **工具失败**：某个工具返回了乱码或挂了，能不能不让整个任务崩？
- **审计追溯**：两周前的某次执行跑飞了，能不能回放当时到底发生了什么？

一个人写到第 3 个要求就放弃了。所以就有了**通用 Agent 框架** —— 它们在骨架之外提供了**工程化的基础设施**，让你只写业务逻辑，不用自己造轮子。

本节我们讲三个代表：

- **Openclaw**：工程化可控的通用 Agent 平台，强在架构分层
- **Codex**：OpenAI 的代码 / 终端 Agent
- **Claude Code**：Anthropic 的终端 + IDE Agent（我们现在用的就是它）

## 2. 通用 Agent 的 3 个关键特性

通用 Agent 和"手写循环"的本质差距，在于它提供了三个可复用的高阶能力。

### 2.1 Skills（技能）：按需挂载的能力包

<img src="/static/images/ai-office/skills-subagents.svg" alt="Skills 与子 Agent" style="width:100%;max-width:900px;"/>

**Skill 是什么？**

一个 Skill 就是一个**可复用的领域知识包**，通常是一个目录，里面有一个 `SKILL.md`（说明书）加上一些脚本 / 模板 / 示例。

`SKILL.md` 告诉 Agent 三件事：

1. **什么时候应该用这个 Skill**（触发词 / 触发条件）
2. **这个 Skill 里面装的步骤 / 约定**
3. **可以调用哪些工具、按什么格式产出**

**Skill 怎么被加载？**

当用户 prompt 里出现触发词（或者 Agent 在推理时判断"这事儿应该用这个 Skill"），框架会**自动把对应 Skill 的 `SKILL.md` 并入当前系统提示**。Agent 拿到额外的知识和约束，就能按这个 Skill 的范式去工作。

**举个本仓库的例子**：

```
skills/
├── pptx/          # 做 PPT 的 Skill
├── xlsx/          # 做 Excel 的 Skill
├── docx/          # 做 Word 的 Skill
├── pdf/           # 做 PDF 的 Skill
└── scf-platform-context/   # 本仓库的业务 framing Skill
```

当你在 Claude Code 里说"帮我做一份 PPT"，`pptx` Skill 会被自动挂载，Agent 就知道用 `python-pptx` 库、按特定的代码范式生成 `.pptx` 文件。

**Skill 的核心价值**：

- **把领域 know-how 从 prompt 里抽出来**，不用每次都复述
- **按需加载**，不占平时的上下文
- **团队共享**，你写的好 Skill 全公司受益

> **讲课提示**：这里可以现场打开 `skills/pptx/SKILL.md` 给学员看，让他们对"Skill 不是什么神秘的东西，就是一份说明书"有直观感受。

### 2.2 Sub-agent（子 Agent）：派出去的专项小组

**子 Agent 是什么？**

子 Agent 是**主 Agent 派出去处理子任务的独立执行体**。关键词是"独立" —— 它有自己的上下文，做完只把结论回传给主 Agent。

打个比方：主 Agent 是团队负责人，子 Agent 是他派去做调研的下属。下属跑去查了一堆资料，最后只给负责人一份**摘要报告**，负责人不需要看下属读过的每一份文件。

**子 Agent 的核心价值**：

| 价值 | 对应的问题 |
|---|---|
| 保护主上下文 | 子任务本来要读 100 个文件，这些文件如果都进主 Agent 上下文，会把窗口撑爆。子 Agent 读完只回一个 300 字摘要，主 Agent 上下文干净 |
| 并行 | 可以同时派 3 个子 Agent 做 3 件独立的调研，比串行快得多 |
| 专业化 | 不同子 Agent 可以配不同的工具权限 / 不同的模型 |

**在 Claude Code 里的具体实现**：

- `Explore` 子 Agent：专门做"探索式代码搜索"，会读很多文件但只回结构化摘要
- `Plan` 子 Agent：专门做"设计实现方案"，不改代码，只产出计划
- 自定义子 Agent：可以根据项目需要写在 `.claude/agents/` 目录下

**Skill vs 子 Agent 的区别**：

- **Skill 改变"主 Agent 怎么想"**（注入知识、规则、工具使用规范）
- **子 Agent 改变"谁来做这件事"**（把任务委派给独立上下文的执行体）

两者不是替代关系，而是**经常一起用**：主 Agent 在某个 Skill 指导下决定要做调研，于是派一个 Explore 子 Agent 去做。

### 2.3 MCP / Tool 生态

**MCP（Model Context Protocol）** 是 Anthropic 主导的一套"Agent 和外部系统怎么对接"的开放协议。

简单来说，它解决的问题是：

> **每个外部系统（Jira、Slack、Gmail、本地文件、数据库……）都得给 Agent 开一个口子，怎么做到标准化？**

过去每家 Agent 框架都自己写一套工具对接层，换个框架全部白写。MCP 把这套对接约定**标准化**了：

- 一个外部系统只需实现一个 MCP Server
- 任何支持 MCP 的 Agent 框架（Claude Code、Codex 的部分版本、未来更多）都能直接接入

结果就是：**你为 Jira 写一次 MCP Server，所有 Agent 框架都能用你的 Jira**。这和 USB 统一了所有硬件接口是一个意思。

## 3. Openclaw 5 大层架构（重点）

<img src="/static/images/ai-office/openclaw-5layers.svg" alt="Openclaw 5 层架构总览" style="width:100%;max-width:900px;"/>

Openclaw 的工程亮点在于它把 Agent 拆成了 **5 个清晰的层**，每一层职责单一、可替换、可独立升级。这 5 层我们逐层过。

### 3.1 Layer 1 · 接口与输入层（感官）

**作用**：实现多渠道接入，把外部请求作为统一事件送入系统。

**支持的输入渠道**：

| 渠道 | 场景 |
|---|---|
| **CLI（终端命令行）** | 最核心的输入方式，面向开发者和内部运营 |
| **WhatsApp / Telegram** | 个人端 IM 接入，面向终端用户 |
| **Discord / Slack** | 团队协作场景 |
| **Cron（定时任务）** | 无人值守的定时触发，如"每天早上 8 点跑一次" |
| **Webhooks（回调）** | 由外部事件触发，如"有人在 GitHub 提了 PR 就跑" |

**设计要点**：每个渠道都是一个**独立的适配器**，互不影响；新增一个渠道（例如企业微信）只要加一个适配器，**不需要动下游任何一层**。

### 3.2 Layer 2 · 网关控制平面（交通枢纽）

<img src="/static/images/ai-office/layer2-lanes.svg" alt="Layer 2 车道队列" style="width:100%;max-width:900px;"/>

**作用**：标准化 + 隔离 + 并发控制。这是 Openclaw 最核心的工程亮点之一。

包含三大组件：

#### (1) 通道适配器 —— 输入归一化

不同渠道来的消息格式不同（WhatsApp 消息 vs CLI 命令 vs Webhook JSON），通道适配器把它们**统一转换成标准结构**。

结果：下游完全不需要关心"这条消息从哪来的"，**接入层和业务层彻底解耦**。

#### (2) 会话路由器 —— 会话隔离

为每个用户 / 每个任务创建一个**独立的会话**。每条进来的事件，根据用户 id / 任务 id 被路由到对应会话。

目的：**彻底避免跨会话的状态污染和上下文串扰**。用户 A 的对话历史不会进到用户 B 的推理里。是支持多用户、多任务并行的基础。

#### (3) 车道队列 —— 并发控制

这是最精妙的一环。

**核心设计**：为每个独立会话分配一条**专属车道**。同一条车道内的事件**严格 FIFO 串行执行**——前一个事件处理完成后，才会处理下一个事件。

**为什么这么设计？**

考虑一个场景：用户 A 连续发了 3 条消息——"查天气"、"发邮件"、"写总结"。如果这三条并发进入同一个 Agent：

- 它们可能同时修改会话的对话历史
- 可能同时读写同一块内存
- 可能同时调用同一个外部工具

**结果就是经典的并发 bug** —— 竞态条件、状态撕裂、执行顺序不可预测。

而 Openclaw 的车道串行化，**从根源上消除**了这类并发问题：

- 同一会话内永远只有一个事件在跑
- 执行顺序完全可预测
- 故障可复现

同时**不同车道之间是并行的**——用户 A 在跑的时候，用户 B、C 的车道照样飞速处理，不会互相阻塞。

> **讲课提示**：车道这个概念很多后端工程师听过"信箱"（actor model）就懂了，本质是一样的——把并发问题降维成"每个实体内部串行、实体之间并行"。可以对比"银行柜台"：每个柜台一条队列，你不会和别人的业务混在一起。

### 3.3 Layer 3 · Agent 运行器（大脑）

**作用**：任务的理解、拆解、规划与上下文管理。所有决策都在这里生成。

包含三大组件：

#### (1) 模型解析器 —— LLM 选型与调度

**作用**：屏蔽具体模型，实现模型层的完全解耦。

支持的模型：

- **Claude**（Anthropic）
- **OpenAI** GPT 系列
- **本地 Ollama** 部署的各种开源模型

**动态选型**：根据任务特点选最合适的模型。比如：

- 简单的文件重命名 → 本地轻量模型（省钱）
- 复杂的任务规划 → Claude Opus（最强推理）
- 大批量文本处理 → Gemini Flash（便宜 + 大上下文）

业务代码完全不需要知道底层用的是哪个模型，换模型只需改配置。

#### (2) 系统提示词构建器 —— 上下文动态注入

**作用**：把散落在各处的信息**拼装成一份高质量的系统提示词**。

输入的原料：

- 任务的核心要求（用户 prompt）
- 工具能力说明（工具清单）
- 内存里的历史信息（MEMORY.md、相关向量检索结果）
- 用户的个性化规则（Skills 里的约定）

输出：一份完整的 `system prompt`，塞给 LLM。

这是 Agent 规划能力的**核心载体**。系统提示词质量高，Agent 发挥就稳；反之 Agent 很容易跑偏。

#### (3) 上下文窗口防护 —— Token 精细化管理

<img src="/static/images/ai-office/layer3-context-guard.svg" alt="上下文窗口防护：可控降级 vs 静默截断" style="width:100%;max-width:900px;"/>

**这是解决 Agent 长周期任务失败的关键设计。**

常见的低质 Agent 实现：对话历史一旦超过上下文窗口，**从头开始默默截断**。结果：系统指令先被截，然后是任务要求——最关键的信息丢了，Agent 完全跑飞，且**你不知道它什么时候跑飞的**。

Openclaw 的做法：

1. **提前计算 Token 占用量**（不是等溢出了才处理）
2. 按优先级保留：
   - **必保留**：系统指令、任务要求、最近的执行结果、当前工具输出
   - **压缩**：早期/中期对话 → 摘要成 1-2 段关键结论
   - **丢弃**：冗长的原始工具输出 → 只留有用字段
3. **可控降级，而非静默截断**：整个过程都有日志记录

工程效果：长周期任务（几小时甚至一整天）不会因为上下文溢出而无声跑飞。

### 3.4 Layer 4 · 执行与工具层（双手）

**作用**：把大脑的决策转化为真实操作。这是 Openclaw "工具优先设计"理念的体现。

包含两大模块：

#### (1) LLM API 调用模块

统一封装了不同大模型的调用接口，提供：

- **流式响应**：token 逐个吐出来，体验不卡顿
- **错误重试**：网络抖动自动重试，不让整个任务挂
- **成本统计**：每次调用的 token 和花费都记下来
- **异常兜底**：模型挂了 / 限流 / 鉴权失败都有明确处理

**这一层是 Agent 运行器和底层大模型之间的桥梁。**

#### (2) 沙箱运行时

工具执行的核心载体。虽然叫沙箱，但它**开放了三大核心执行能力**：

| 能力 | 底层 | 典型用途 |
|---|---|---|
| **Shell 命令执行** | Bash / Zsh | 代码运行、环境配置、文件批处理、系统操作 |
| **无头浏览器** | Puppeteer | 网页操作、数据爬取、页面交互、截图 |
| **文件系统访问** | 本地 fs | 文档处理、代码修改、数据归档、报表生成 |

这三样加起来，Agent 就能完成**几乎所有办公自动化任务** —— 写代码、跑脚本、上网查东西、改文档、做报表。

### 3.5 Layer 5 · 混合内存系统（长期记忆）

<img src="/static/images/ai-office/layer5-memory.svg" alt="混合内存系统三层分工" style="width:100%;max-width:900px;"/>

**作用**：解决 Agent 最常见的"失忆"问题，是长周期任务的基础。

**三层混合存储**：

#### (1) JSONL 日志 —— 原始记录

- **形态**：每行一条 JSON 事件，append-only 持久化
- **内容**：每一次的用户输入、模型规划、工具调用、执行结果、异常信息、token 统计
- **用途**：审计追溯、故障回放、做后续摘要/索引的原料
- **类比**：行车记录仪——全量记录，事后查

#### (2) MEMORY.md —— 核心规则与知识

- **形态**：结构化的 Markdown 文件，按主题 / 类型分块
- **内容**：用户偏好（"总是用 Python 做脚本"）、团队规则（"禁止 force push"）、领域知识（"数据库 schema"）
- **用途**：每一轮都**自动注入系统提示词**，是跨会话、跨任务的持久规则
- **类比**：员工手册——始终挂在墙上，干什么事都得照着来

**关键点**：MEMORY.md 避免了"全量记录"带来的 token 浪费 —— 你不需要把所有对话都记着，只记下**影响未来决策的核心规则**就够了。

#### (3) 向量 / FTS5 索引 —— 精准语义检索

- **形态**：向量嵌入 + 全文索引
- **索引来源**：历史 JSONL 日志、用户文档、代码、内部知识库
- **用途**：**按当前任务"按需召回"相关历史** —— 不是全量塞进上下文，而是检索最相关的几段
- **类比**：图书馆搜索引擎——用到再查

**三层协同的效果**：

- 日志层：全量可追溯（知道发生过什么）
- 规则层：始终在场（知道该怎么做）
- 索引层：按需查询（知道去哪找更多细节）

长周期任务从此不再"失忆"——因为它有了**合适层级的记忆**。

## 4. Openclaw / Codex / Claude Code 横向对比

<img src="/static/images/ai-office/agents-compare.svg" alt="三者横向对比" style="width:100%;max-width:900px;"/>

| 维度 | Openclaw | Codex（OpenAI） | Claude Code（Anthropic） |
|---|---|---|---|
| **定位** | 工程化可控的通用 Agent 平台 | 面向开发者的代码/终端 Agent | 面向研发全流程的交互式 Agent |
| **核心强项** | 5 层架构 / 会话隔离 / 混合内存 | 深度 OpenAI 生态 / GPT 能力 | Skills / 子 Agent / MCP 丰富 |
| **典型场景** | 多用户企业内部 Agent 平台 | 代码生成 / 终端任务 | 研发全流程 / 文档 / 数据处理 |
| **可控性** | 高（工程化导向） | 中 | 中 |
| **多用户并发** | 原生支持（车道队列） | 需要自行解决 | 需要自行解决 |
| **模型可换性** | 强（模型解析器） | 基本锁定 OpenAI | 基本锁定 Anthropic |
| **上下文管理** | 主动防护（精细化） | 标准 | 大窗口 + 摘要 |
| **学习曲线** | 中（要理解分层） | 低（装完就能用） | 低（装完就能用） |

**选型建议**：

- **需要做内部 Agent 平台、要给多个人/团队用** → Openclaw
- **纯个人写代码 / 跑终端任务、深度绑 OpenAI 生态** → Codex
- **个人或小团队做研发全流程辅助、要丰富的 Skills 生态** → Claude Code

> **讲课提示**：不要把三者说成"谁好谁坏"。他们定位不同，受众不同，好比"货车 vs 轿车 vs SUV"。重点是让学员理解"用对场景"。

## 5. 同一个任务的三种写法（示例）

任务：**"帮我整理今天的项目日报，格式不限"**

**Openclaw 典型流程**（假设部署在企业内部）：

```
1. 你在 IM 群里 @Openclaw "整理一下今天的日报"
2. Layer 1（IM 适配器）收到消息
3. Layer 2 路由到你的专属车道
4. Layer 3 的系统提示词构建器：
   - 注入你的 MEMORY.md（里面有"日报固定格式"规则）
   - 从向量索引里捞出本周项目进展
5. Layer 4 调用工具：git log、Jira API、Slack API
6. Layer 5 把这次执行记录到 JSONL 日志
7. IM 里回复你一份日报，全程 30 秒
```

**Codex 典型流程**：

```bash
$ codex "帮我整理今天的项目日报"
> 调用 web_search 查看今日动态...
> 读本地 notes.md...
> 生成日报...
```

产出一段 Markdown，你 copy-paste 到需要的地方。

**Claude Code 典型流程**：

```
$ claude "帮我整理今天的项目日报"
> [自动挂载 docx Skill]
> [派 Explore 子 Agent 调研本周代码变更]
> [子 Agent 回报调研摘要]
> [生成日报 + 可选生成 .docx 文件]
```

产出可以是 Markdown 或 Word 文件，并且会记住你偏好的格式（下次不用再说）。

**三种产出的差异**：

- Openclaw：企业级、可追溯、多系统整合
- Codex：轻量、快速、代码友好
- Claude Code：交互丰富、产物多样、记忆持久

## 6. 常见误区

### 误区一：Claude Code 只能写代码

**错。** Claude Code 已经有成熟的文档、数据、办公类 Skills（pptx / docx / xlsx / pdf 都是官方 Skills），可以完成绝大多数办公任务。"代码"只是它**最早被证明好用**的场景，不是它的边界。

### 误区二：Skills 越多越好

**错。** 每个 Skill 加载后都会**占用上下文**。大量冗余 Skill 挂着会稀释模型的注意力。正确做法是：

- Skill 要写好**触发词**，让框架只在需要时挂载
- 按"业务域 / 高频场景"来组织 Skill，不要每个小事都开一个 Skill

### 误区三：Openclaw 比 Claude Code "更先进"

**错。** 它们**设计目标不同**，不是同一个维度上的竞争。Openclaw 强在工程化分层（多用户 / 多团队 / 审计 / 可控），Claude Code 强在交互式生产力（个人开发者的"超级搭档"）。在各自场景里都是最合适的选择。

### 误区四：Openclaw 的车道队列会让系统变慢

**错。** 车道是**按会话**分的。一个用户串行，100 个用户就有 100 条并行车道。对单用户而言执行顺序可预测，对系统整体而言照样高并发。
