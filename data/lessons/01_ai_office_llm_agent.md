## 1. 开场：一个一分钟能讲完的类比

想象你请了一个**超级博学但被关在房间里的人**。

- 他读过人类写过的几乎所有书、所有论文、所有代码。
- 但他没有手机，没有电脑，也看不见外面的世界。
- 你只能从门缝塞进去一张纸条，他也只能通过门缝塞出一张纸条回答你。

这个"被关在房间里的天才"就是 **LLM 大模型**。

然后你给他加了三样东西：

1. 一把**电话**，他可以打出去问别人问题。
2. 一台**电脑**，他可以读写文件、执行代码。
3. 一个**笔记本**，他可以把过程记下来，下一次接着做。

同时你告诉他：
"每次你做完一件事，把结果写在纸条上塞回来，我再告诉你下一步。但你如果发现这件事需要好几步，你也可以**自己循环几次**再交卷。"

这样被改造过的天才，就是 **Agent**。

> **讲课提示**：类比不要讲得太久，1-2 分钟打住。目的是让零基础的听众建立"LLM ≠ ChatGPT App"的意识。很多人把"ChatGPT 产品"当成 LLM 本身，其实 ChatGPT 已经是一个 Agent 了。

## 2. 大模型是怎么"听懂"我们的

### 2.1 它其实只做一件事：预测下一个 token

<img src="/static/images/ai-office/llm-nexttoken.svg" alt="LLM = 下一个 token 的概率预测器" style="width:100%;max-width:900px;"/>

你输入 `"今天天气真"`，LLM 的内部工作流程大致是：

1. **分词（tokenization）**：把文字切成一个个 token。你看到的"今天天气真"，在模型眼里可能是 `["今天", "天气", "真"]` 三个 token，每个 token 对应一个整数 id。
2. **过 Transformer**：几十层的神经网络对这些 token 做注意力计算，最后吐出一个巨大的概率分布。这个分布覆盖了词表里的**所有可能的下一个 token**。
3. **采样**：从这个分布里按概率抽一个 token 出来（可以贪心选最高概率的，也可以带随机性）。抽到的 token 就是它"说出来的下一个字"。
4. **追加回输入，继续循环**：把新 token 拼回原始输入，再预测下一个 token，直到撞到停止符（比如句号 / 换行 / EOS）。

换句话说：

> **模型每一步都只干一件事 —— 估计"下一个字最可能是什么"，然后吐出来。**

你看到的一段流畅文字，是**几百次"下一字预测"串起来**的结果。

### 2.2 那它为什么看起来"懂"

因为它在几千亿参数里，把人类写过的大量文字**高度压缩**了一遍。

- 它知道 `"今天天气真"` 后面接 `"好"` 的概率远高于接 `"闷"`、`"冷"` 或 `"梨"`。
- 它知道一段 Python 代码里，`def` 后面通常跟函数名和左括号。
- 它知道一封商务邮件的结尾通常是 `"此致敬礼"` 而不是 `"晚安"`。

这些**统计规律**在它眼里是连续的向量空间，所以看起来它"懂语义、懂语法、懂常识"。但它并不是真的"理解"，它只是**在巨大的语言世界里找到了最可能的下一步**。

### 2.3 那它为什么会"胡编"（hallucinate）

因为它的优化目标是 **"像人说的话"**，而不是 **"真实世界里发生过的事"**。

当你问它一个它不知道的事实（比如"某公司 2024 年 Q1 的营收"），它会继续做它擅长的那件事——**生成一段"看起来像真的"的文字**。这段文字的结构是对的、语气是专业的，但具体数字可能是它随手编的。

这就是为什么：

- 纯 LLM **不适合直接回答需要最新事实的问题**。
- 必须给它"工具"去查真实数据，它才能可靠。
- 这就引出了我们要讲的 Agent。

> **讲课提示**：很多听众会问"那 GPT 怎么能回答我的问题？" 你可以解释：**ChatGPT 已经是一个 Agent**，它背后有 web 搜索、代码解释器、文件读取等一堆工具，不是"纯 LLM"。这节的重点是把**纯模型** 和 **加了工具的模型**分开讲。

## 3. 从 LLM 到 Agent：加上"工具 + 循环"

<img src="/static/images/ai-office/agent-loop.svg" alt="Agent = LLM + Tools + Loop" style="width:100%;max-width:900px;"/>

Agent 的公式只有一行：

```
Agent = LLM + Tools + Loop
```

三个元素缺一不可。

### 3.1 工具（Tools）

**工具是 Agent 的"手脚"**。你给它一份工具清单（每个工具包括：名字、功能描述、参数说明），它在推理时会**挑一个合适的工具**并**填好参数**。

常见工具分几类：

| 工具类别 | 例子 | 用途 |
|---|---|---|
| 网络 | `web_search`、`http_get` | 查实时信息 |
| 文件系统 | `read_file`、`write_file`、`list_dir` | 读写本地文件 |
| Shell | `bash`、`powershell` | 执行系统命令 |
| 数据库 | `sql_query` | 查企业数据 |
| 第三方 API | `send_email`、`create_jira` | 操作外部系统 |
| 代码解释器 | `python_run` | 跑代码、做数据分析 |

一段典型的 Agent 工具调用长这样（伪代码）：

```python
# 模型输出了一段结构化的"工具调用请求"
{
  "tool": "search_weather",
  "arguments": {
    "city": "北京",
    "date": "2026-04-15"
  }
}
```

程序拿到这个 JSON，真的去调了查天气的 API，拿回 `"晴，25°C"`，再**把这个结果塞回给 LLM**，让 LLM 继续下一步推理。

### 3.2 循环（Loop）

**工具返回的结果进入下一轮推理**，这就是循环。每一轮推理，LLM 可以：

- 决定再调一个工具
- 决定根据已有信息直接写答案
- 决定任务已完成，结束

循环的**停止条件**通常有四种：

1. **任务完成**：LLM 自己判断"这活儿干完了"。
2. **步数上限**：为了防止死循环，人为设定一个最大步数（比如 20 步）。
3. **连续失败**：工具调用连续错几次，保护性退出。
4. **用户中断**：人工喊停。

### 3.3 一个最小 Agent 的伪代码

```python
def run_agent(user_task):
    messages = [
        {"role": "system", "content": "你是一个能调用工具的助手。"},
        {"role": "user", "content": user_task}
    ]
    tools = [weather_tool, email_tool]

    for step in range(MAX_STEPS):
        # 调用 LLM，让它决定下一步
        response = llm.chat(messages=messages, tools=tools)

        if response.is_final_answer:
            return response.content   # 任务完成

        # 模型要求调用一个工具
        tool_name = response.tool_call.name
        tool_args = response.tool_call.arguments
        tool_result = execute_tool(tool_name, tool_args)

        # 把工具结果塞回对话，继续循环
        messages.append({"role": "assistant", "content": response.raw})
        messages.append({"role": "tool", "name": tool_name, "content": tool_result})

    raise RuntimeError("达到最大步数仍未完成任务")
```

这段 50 行的伪代码，就是**所有 Agent 框架的核心骨架**。Openclaw、Codex、Claude Code、LangChain，都是在这个骨架上加了大量工程能力（会话隔离、内存管理、错误重试、并发控制……），但骨架不变。

> **讲课提示**：这段伪代码强烈建议你在白板上边写边讲。很多人觉得 Agent 很神秘，看到这 50 行会有"原来就这？"的顿悟感——这个顿悟感是本节最重要的产出。

## 4. LLM 与 Agent 能力对比

<img src="/static/images/ai-office/llm-vs-agent.svg" alt="纯 LLM 与 Agent 能力对比" style="width:100%;max-width:900px;"/>

| 能力 | 纯 LLM | Agent |
|---|---|---|
| 回答单轮问题 | ✓ | ✓ |
| 根据训练数据生成文本 | ✓ | ✓ |
| 查实时信息（天气、股价、新闻） | ✗ | ✓（用 web 工具） |
| 读写本地文件 | ✗ | ✓（用文件系统工具） |
| 操作电脑 / 终端 | ✗ | ✓（用 shell 工具） |
| 多步推理 + 失败重试 | ✗ | ✓（循环） |
| 跨会话记住用户偏好 | ✗ | ✓（用内存系统） |
| 调用 API / 数据库 | ✗ | ✓（用工具） |
| 执行代码 | ✗ | ✓（用代码解释器工具） |

一句话总结：

> **LLM 是脑子，Agent 是带着脑子 + 工具 + 记忆 + 循环的执行体。**

## 5. 常见误区（重点讲一讲，学生最容易混淆这三条）

### 误区一：模型越大，Agent 就越强

**错。**

Agent 的强弱更多取决于**工具设计**和**上下文管理**，而不是模型本身。用同一个 Claude Opus 做 Agent，工具清单写得烂，产出照样差；反之，给一个中等模型配一套好用的工具 + 规则，反而能超过"大模型裸用"。

实际项目里，选模型和设计工具大概是 3:7 的重要性。

### 误区二：有了 Agent 就能全自动化

**错。**

Agent 在长周期任务里容易出现三种问题：

1. **幻觉累积**：第 3 步基于第 2 步错误的结论继续推，结果越走越偏。
2. **上下文溢出**：跑了十几步后对话历史太长，超出上下文窗口被截断，早期的关键指令丢失。
3. **工具失败链**：某个工具返回了非预期格式，模型没处理好，一路错下去。

所以实际工程里都要做**人类干预点**：分阶段让人 review、关键决策让人确认、长任务做断点续跑。

### 误区三：给工具就行了，Prompt 随便写

**错。**

再好的工具，Prompt 写得含糊，Agent 也不知道**什么时候**该调用这个工具。你需要在 system prompt 里把"在什么情况下应该用哪个工具"写清楚，才能稳定发挥。

这一点在下一节会详细讲。

## 6. 一个具体的完整例子：查天气 → 发邮件

假设我们要让 Agent 完成这样一个任务：

> "查一下北京今天的天气，如果下雨，就发邮件给老板提醒他带伞。"

它在 Agent 循环里会这么走：

**Step 1** — LLM 推理：
"任务是'查天气 + 条件发邮件'。第一步应该先查天气。我有 `search_weather` 工具，调用它。"

```json
{"tool": "search_weather", "arguments": {"city": "北京"}}
```

**Step 2** — 工具执行返回：
`{"weather": "小雨", "temperature": 18}`

**Step 3** — LLM 推理：
"查到是小雨，根据任务要求，应该发邮件。调用 `send_email` 工具。"

```json
{
  "tool": "send_email",
  "arguments": {
    "to": "boss@company.com",
    "subject": "今日北京天气提醒",
    "body": "今天北京有小雨，气温 18°C，记得带伞。"
  }
}
```

**Step 4** — 工具执行返回：
`{"sent": true, "message_id": "..."}`

**Step 5** — LLM 推理：
"邮件发出去了。任务完成。"

返回最终答复：`"已查询天气（小雨，18°C）并发送提醒邮件给老板。"`

整个过程 Agent 做了 **3 次 LLM 调用 + 2 次工具调用**，全自动完成。
