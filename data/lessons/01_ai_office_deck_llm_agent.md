## 从 LLM 到 Agent

本节要回答三个问题：

- **大模型到底在做什么？** 我们每天用 ChatGPT、Claude，但它内部发生了什么？
- **LLM 和 Agent 的本质差别是什么？** 为什么同一个模型，"套壳"前后能力差距巨大？
- **Agent 是怎么一步步解决复杂任务的？** 它是"魔法"吗，还是工程？

讲完这一节，你应该能用一段话讲清大模型的工作原理，画出 Agent 的基本循环，并说出 3 件"纯 LLM 做不到但 Agent 能做"的具体事。

> 本节是整个培训的地基——后面讲 Openclaw、讲 Prompt，都是在这个地基上盖楼。

---

## 一个类比：关在房间里的博学者

想象你请了一个**博学但被关在房间里的人**：

- 他读过人类写过的几乎所有书、所有论文、所有代码
- 但他没有手机、没有电脑，看不见外面的世界
- 你只能从门缝塞纸条进去，他也只能从门缝塞纸条出来回答你

这个被关起来的博学者，就是 **LLM 大模型**——只会接话，不会动手。

然后你给他加了三样东西：

- **一把电话**，他可以打出去问别人问题
- **一台电脑**，他可以读写文件、跑代码
- **一个笔记本**，他可以把过程记下来，下次接着做

这样被改造过的博学者，就是 **Agent**。

> ChatGPT 产品早就不是"纯 LLM"——它背后接了搜索、代码解释器、文件读取等一整套工具。我们平时说的"用 AI"，大多数时候其实在用 Agent。

---

## LLM 内部在做什么：预测下一个 token

<img src="/static/images/ai-office/llm-nexttoken.svg" alt="LLM 下一个 token 预测" />

你输入 `"今天天气真"`，LLM 内部走了四步：

- **① 分词（Tokenizer）**：把文字切成一个个 token，比如 `["今天", "天气", "真"]`
- **② 过 Transformer**：几十层神经网络计算后，吐出一个概率分布，覆盖词表里所有可能的下一个字
- **③ 采样**：按概率抽一个 token 出来，这就是它"说出来的下一个字"
- **④ 拼回输入，继续循环**：把新 token 接回原文，再算下一个，直到遇到停止符

整个过程，**模型每一步只做一件事**——估计"下一个字最可能是什么"。你看到的一整段流畅文字，是几百次"预测下一字"串起来的结果。

> 记住这一句：**LLM 不是在思考，是在预测**。

---

## 为什么 LLM 看起来"懂"

模型有几千亿个参数，训练时"读过"了人类写过的大量文字。这些文字里的规律被高度压缩进了参数：

- 它知道 `"今天天气真"` 后面接 `"好"` 的概率远高于 `"梨"`
- 它知道 Python 代码里 `def` 后面通常跟函数名和左括号
- 它知道商务邮件的结尾一般是 `"此致敬礼"` 而不是 `"晚安"`

这些**统计规律**在它内部表现为连续的向量空间，所以看起来它"懂语义、懂语法、懂常识"。

但它并没有真正"理解"你——它只是在一个巨大的语言世界里，**找到最可能的下一步**。

> 这就是为什么它能写诗、写代码、写方案——因为人类在各个领域都留下了大量文字可供模仿。

---

## 为什么 LLM 会"胡编"

LLM 的训练目标只有一个：**让它说出来的话，像人说的**。

它的目标**不是**"真实世界里发生过的事"。所以当你问它一个它不知道的事实：

- 它不会说"我不知道"
- 它会继续做它擅长的事——生成一段"看起来像真的"的文字
- 这段文字结构正确、语气专业，但数字、链接、引用可能是它随手编的

比如你问"某公司 2024 年 Q1 营收"，它可能给你一个精确到两位小数的数字，看起来非常可信——但实际上可能完全错。

这种现象叫 **hallucination（幻觉）**。

> 纯 LLM 不适合直接回答"需要事实"的问题。想让它可靠，就必须给它**工具**去查真实数据。

---

## 纯 LLM 解决不了的办公问题

你想让 AI 帮你做这些事，纯 LLM 全都做不到：

- **"查一下北京今天的天气，如果下雨就提醒老板"**——它既不能查实时天气，也不能发邮件
- **"读一下这份 Excel，把异常数据挑出来"**——它不能读你本地的文件
- **"把这段代码跑一下，看看会不会报错"**——它不能执行代码
- **"记一下我们公司总是用 Python 写脚本，以后别推荐 Go"**——它跨会话就忘了

这些任务的共同点：**需要和外部世界交互**。纯 LLM 被关在房间里，没有手，没有眼睛，没有记忆。

> 所以我们需要给它装上手、眼睛、记忆——这个**装配过程**，就叫做 Agent。

---

## Agent 的公式

<img src="/static/images/ai-office/agent-loop.svg" alt="Agent = LLM + Tools + Loop" />

Agent 的公式就一行：

> **Agent = LLM + Tools + Loop**

三个元素缺一不可：

- **LLM**：还是那个博学者，负责理解你的意图、决定下一步做什么
- **Tools（工具）**：外部能力的入口，让它能查网、读文件、执行代码、发邮件
- **Loop（循环）**：工具返回的结果会送回给 LLM，让它基于结果决定下一步，形成闭环

这三样加起来，LLM 就从"只会接话"变成了"能做事"。

> 所有 Agent 框架——Openclaw、Codex、Claude Code、LangChain——本质都是这三件事。

---

## 什么是"工具"

工具就是**外部能力的入口**。框架给 LLM 一份工具清单（名字、功能描述、参数说明），LLM 在推理时会选一个合适的工具并填好参数。

常见工具大致分六类：

- **网络类**：`web_search`、`http_get`，用来查实时信息
- **文件系统类**：`read_file`、`write_file`，用来读写本地文件
- **Shell 类**：`bash`，用来执行系统命令
- **数据库类**：`sql_query`，用来查企业内部数据
- **第三方 API 类**：`send_email`、`create_jira`，用来操作外部系统
- **代码解释器类**：`python_run`，用来跑代码做分析

LLM 调用工具的输出长这样（一段结构化的 JSON）：

```json
{
  "tool": "search_weather",
  "arguments": {"city": "北京", "date": "2026-04-15"}
}
```

程序拿到这段 JSON，真的去调了天气 API，拿回 `"晴, 25°C"`，再塞回 LLM 让它继续推理。

> 工具设计的好不好，比模型大不大更决定 Agent 的表现。

---

## 什么是"循环"——以及它什么时候停

LLM 每调一次工具，返回的结果会送回给它做下一轮推理。每一轮，LLM 可以：

- 决定再调一个工具（还需要更多信息）
- 决定根据已有信息直接写答案（任务完成）
- 决定任务不可能完成（放弃并报告）

循环在以下四种情况停止：

- **任务完成**：LLM 自己判断"够了，可以交卷"
- **达到最大步数**：人为设一个上限（比如 20 步），防止死循环
- **连续失败**：工具调用接连出错，保护性退出
- **用户中断**：人工喊停

> 停止条件是保险丝。没有它，Agent 可能陷入自言自语或反复调用工具的死循环。

## 最小 Agent 伪代码

Agent 的核心骨架只需要 50 行。把它吃透，你就明白所有框架都在这个骨架上加工程能力：

```python
def run_agent(user_task, tools):
    messages = [
        {"role": "system", "content": "你是一个能调用工具的助手"},
        {"role": "user", "content": user_task}
    ]

    for step in range(MAX_STEPS):
        # 调 LLM，让它决定下一步
        response = llm.chat(messages=messages, tools=tools)

        if response.is_final_answer:
            return response.content   # 任务完成

        # LLM 要求调用某个工具
        tool_result = execute_tool(
            response.tool_call.name,
            response.tool_call.arguments
        )

        # 把工具结果塞回对话，进入下一轮
        messages.append({"role": "assistant", "content": response.raw})
        messages.append({"role": "tool", "content": tool_result})

    raise RuntimeError("超过最大步数仍未完成")
```

看完这段你应该有一种"原来就这？"的感觉——是的，**Agent 的本质就这么简单**。后面讲的 Openclaw / Codex / Claude Code，都是在这个骨架外面包了工程化的壳。

## LLM vs Agent 能力对比

<img src="/static/images/ai-office/llm-vs-agent.svg" alt="LLM 与 Agent 能力对比" />

同一个模型，**外面有没有套工具 + 循环**，能做的事差距巨大：

| 能力 | 纯 LLM | Agent |
|---|---|---|
| 回答单轮问题 | ✓ | ✓ |
| 查实时信息（天气、股价） | ✗ | ✓ |
| 读写本地文件 | ✗ | ✓ |
| 执行代码 / Shell 命令 | ✗ | ✓ |
| 多步推理 + 失败重试 | ✗ | ✓ |
| 跨会话记住用户偏好 | ✗ | ✓ |

> LLM 是大脑；Agent 是带着大脑 + 工具 + 记忆 + 循环的执行体。

## 一个完整例子：查天气 → 发邮件

任务：**"查一下北京今天的天气，如果下雨，就发邮件给老板提醒带伞"**

Agent 在循环里大概这么走：

- **Step 1**——LLM 推理："第一步应该查天气"，输出 `{"tool": "search_weather", "args": {"city": "北京"}}`
- **Step 2**——工具执行返回 `{"weather": "小雨", "temperature": 18}`
- **Step 3**——LLM 推理："是小雨，要发邮件"，输出 `{"tool": "send_email", "args": {"to": "boss@...", "subject": "天气提醒", "body": "今天小雨 18°C 记得带伞"}}`
- **Step 4**——工具执行返回 `{"sent": true}`
- **Step 5**——LLM 判断："任务完成"，返回 `"已查询天气并发送提醒邮件"`

整个任务 Agent 自主完成了 **3 次 LLM 调用 + 2 次工具调用**，你只说了一句人话。

> 这就是 Agent 的价值——把"一个人要做半小时的流程"压成一次对话。

## 三个常见误区

**误区一：模型越大，Agent 就越强**

错。Agent 的强弱更多取决于**工具设计**和**上下文管理**，而不是模型本身。实际经验：选模型和设计工具的影响力大概是 **3:7**。

**误区二：有 Agent 就能全自动化**

错。长周期任务会踩三个坑：

- **幻觉累积**：第三步基于第二步的错误结论继续推
- **上下文溢出**：跑十几步后历史太长，关键指令被截断
- **工具失败链**：某个工具格式异常，模型处理不好，一路错下去

所以实际工程里都要做**人类干预点**，不是全自动。

**误区三：给工具就行，Prompt 怎么写无所谓**

错。再好的工具，Prompt 没写清"什么情况下用哪个工具"，Agent 也不知道什么时候该调用。Prompt 和工具是一起设计的，不能分开。

> Agent 不是魔法，是工程。它的效果取决于你把每个环节设计得够不够细。

---

## 小结 + 过渡到第 2 节

这一节的三个核心观点：

- **LLM 只做一件事**：预测下一个 token
- **Agent 的公式**：LLM + Tools + Loop
- **50 行就能写一个 Agent**：所有框架都是在这个骨架上加工程能力

但是——**当你真的要把 50 行的 Agent 拿去给整个公司用**，马上会撞上一堆问题：

- 10 个人同时用，他们的对话历史会不会串？
- 今天想换个模型试试，代码要全部重写吗？
- 跑了 2 小时的任务，上下文早就溢出了，它还记得前面做过什么吗？
- 工具返回乱码或挂了，整个任务会不会直接崩？
- 两周前的某次执行跑飞了，能查到当时发生了什么吗？

这些就是**通用 Agent 框架**要解决的事。
